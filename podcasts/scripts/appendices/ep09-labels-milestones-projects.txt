Now cross-references. This is one of those GitHub features people use constantly without always naming it.

[ALEX]
Very true. A cross-reference is when one GitHub item mentions another in a way GitHub understands and links. Common types include referencing an issue from another issue, referencing an issue from a pull request, and using special closing keywords in a pull request to connect code changes to issues.

[JAMIE]
Let's start with the basic typing pattern.

[ALEX]
The most common form is the number reference, like hash 42, inside the same repository. GitHub turns that into a link to issue or pull request number 42. You can also reference items across repositories using owner slash repo, then hash number.

[JAMIE]
And when you do that, GitHub often creates a visible backlink, right?

[ALEX]
Yes. That's one of the most useful effects. The referenced issue often shows that it was mentioned elsewhere, which helps people trace related work and discussion.

[JAMIE]
Now the special case: the closes keyword.

[ALEX]
Important one. If a pull request description or a merged commit message uses a closing keyword like closes, fixes, or resolves followed by an issue reference, GitHub can automatically close the linked issue when the pull request is merged into the right branch.

[JAMIE]
So not when the PR is opened.

[ALEX]
Correct. The keyword fires on merge, not merely on mention. That's crucial. A plain reference links discussion. A closing keyword links discussion plus intended completion.

[JAMIE]
That's a subtle but important distinction.

[ALEX]
It is. And it's part of writing responsibly on GitHub. If you're not sure the PR fully solves the issue, use a plain reference rather than a closing keyword.

[JAMIE]
What's the compact learning summary here?

[ALEX]
Cross-references connect context. Plain references say, "these items are related." Closing keywords say, "merging this should complete that issue." Use them intentionally.

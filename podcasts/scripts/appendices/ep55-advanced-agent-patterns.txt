[ALEX]
Let’s start with architecture. A single agent can be good at one task, but a more advanced setup uses a parent agent and specialized subagents.

[JAMIE]
What is a subagent in plain language?

[ALEX]
An agent is a standalone AI assistant focused on a domain, like ARIA patterns. A subagent is that same kind of specialist, but invoked by another agent. So a general web accessibility auditor might hand off one part of the job to an ARIA specialist, another to a contrast checker, another to a keyboard navigation specialist, and so on.

[JAMIE]
So the parent agent becomes more of an orchestrator than a do-everything expert.

[ALEX]
Yes. A useful example is a web accessibility wizard receiving a request to audit a page. Instead of doing one broad, shallow pass, it can orchestrate multiple specialists: one checking ARIA roles, one checking color contrast, one checking tab order, one checking images and headings, one checking tables. Then it gathers all of those results into a consolidated audit report.

[JAMIE]
That sounds more reliable than one giant prompt asking one agent to remember every accessibility domain at once.

[ALEX]
That’s the core idea. Delegation over monolithic design.

[JAMIE]
How do you actually tell an agent to use subagents?

[ALEX]
In the source, the pattern is to put that behavior into the agent’s instructions or a custom prompt and invoke subagents through a function like runSubagent. The exact implementation may vary by platform, but conceptually the parent passes the subagent’s name, a targeted prompt, and often a brief description of the task.

[JAMIE]
So something like: call the ARIA specialist, pass in the component HTML, ask for an ARIA compliance review. Then separately call the contrast specialist with CSS rules and ask for color contrast verification.

[ALEX]
Exactly. Small, focused assignments.

[JAMIE]
When should learners reach for subagents instead of one agent?

[ALEX]
Several cases. For a full page audit, a primary web accessibility agent can dispatch all specialists in parallel and then merge the findings. For code review, a fixer agent might call an ARIA specialist and a contrast specialist to verify whether a proposed fix really works across domains. For document auditing, a document accessibility wizard can serve as a single entry point while delegating to platform-specific experts for Word, PDF, or Excel. And for Markdown, a Markdown accessibility assistant might call a link checker to cross-check link text and heading hierarchy.

[JAMIE]
I like that because it mirrors how human teams work. One coordinator, several specialists.

[ALEX]
Yes, and the appendix gives four principles for making that work well. First, delegation over monoliths: one parent calls specialists, but specialists generally should not call other specialists, because that can create runaway loops or tangled control flow. Second, error recovery: if one subagent fails, the parent logs the failure and continues the rest of the audit. Third, context preservation: the parent passes enough context, such as HTML, CSS, or the document content, so the subagent can make grounded judgments. Fourth, result aggregation: the parent is responsible for combining everything into one coherent report.

[JAMIE]
That error recovery point feels especially important in accessibility workflows. You don’t want one broken checker to silence five other useful findings.

[ALEX]
Precisely.

[PAUSE]

[ALEX]
The next big idea is skills.

[JAMIE]
And here the word skill means something more structured than just, “the model is good at a thing,” right?

[ALEX]
Right. In this appendix, a skill is a reusable multi-step workflow packaged with reference files, templates, and automation scripts. You can think of a skill as a repeatable procedure. It can be invoked by an agent, or in some environments directly by a user using a slash command.

[JAMIE]
So instead of re-explaining the whole process every time, you encapsulate the workflow.

[ALEX]
Exactly. For example, a web-scanning skill might include crawling logic, URL inventory templates, browser automation code, and reference documentation. A report-generation skill might include severity scoring algorithms, scorecard templates, CSV export logic, and WCAG mapping data.

[JAMIE]
That sounds a lot like good software engineering: don’t duplicate process, package it.

[ALEX]
Yes, and the appendix lists a broad library of core skills, more than twenty-five, organized by capability.

[JAMIE]
Can we walk through those categories?

[ALEX]
Absolutely. For web scanning, there are skills like web-scanning and web-severity-scoring, used for page inventory, audit scoping, and severity calculation. For web remediation, there are skills like web-issue-fixer and web-csv-reporter, which support applying fixes and exporting findings.

For document scanning, skills include document-scanning and document-accessibility-rules, used to audit Word, Excel, PDF, or EPUB and map issues to WCAG. For document remediation, there are document-remediation and document-csv-reporter, focused on fixing documents and generating compliance reports.

For Markdown, the library includes markdown-accessibility, markdown-fixer, and markdown-csv-reporter, which are useful for auditing Markdown files and fixing link text, heading structure, or things like confusing emoji usage.

There are also CI and CD integration skills such as lighthouse-scanner and github-accessibility-scanner for parsing continuous integration reports and tracking regressions. Framework-specific skills include framework-accessibility, react-patterns, and vue-patterns. Python development tools include python-fact-grounded-coding and pylance-refactoring. Testing-focused skills include testing-coach and accessibility-testing-setup. And compliance skills include wcag-guide, accessibility-rules-reference, and help-url-reference.

[JAMIE]
That’s a strong reminder that accessibility work isn’t just one kind of artifact. It’s websites, documents, Markdown, CI pipelines, framework code, testing setups, and reference guidance.

[ALEX]
Exactly. The skill system reflects the actual breadth of the work.

[JAMIE]
How does a learner use a skill in practice?

[ALEX]
Two main ways are highlighted. One is directly from a chat interface through commands such as slash web-scanning followed by a URL, slash report-generation with findings data, or slash python-fact-grounded-coding with a script name. The other is from within a custom agent’s instructions. In that case, the agent is told when to invoke a skill, what inputs to supply, and what output format is expected.

[JAMIE]
So the agent instructions might say: when you need to audit a webpage, use the web-scanning skill with a URL, use WCAG 2.2 AA as the severity model, and produce JSON.

[ALEX]
Exactly. Clear inputs, clear output expectations.

[JAMIE]
What about writing your own skill?

[ALEX]
The appendix gives a pattern for that too. A custom skill lives in a directory like .github slash skills slash my-skill-name, with a SKILL.md file. That file should define the skill name, its purpose, its inputs, its workflow, its output, and its references.

[JAMIE]
Can you unpack those sections?

[ALEX]
Purpose explains what problem the skill solves. Inputs list what the skill needs, such as a URL, a file path, or optional configuration. Workflow lays out the sequence, like parse input, execute steps, collect results, format output. Output defines the report format, maybe JSON, Markdown, or CSV, and often includes an example. References point to related agents or external documentation.

[JAMIE]
And there are supporting files too.

[ALEX]
Yes. Templates can hold input or output patterns. Scripts can hold automation in shell or Python. Reference data can store lookup tables or compliance mappings.

[JAMIE]
Why is that structure so important?

[ALEX]
Because it makes a skill portable, inspectable, and reusable. And the appendix emphasizes skill reuse across agents. One web accessibility wizard might use web-scanning plus web-severity-scoring. A web issue fixer might reuse web-scanning plus web-issue-fixer. A link checker might reuse web-scanning plus a narrower link-validation subset. The benefit is consistent behavior and less duplicated logic.

[ALEX]
Now let’s move into hooks, which are about lifecycle automation.

[JAMIE]
This is where the system starts doing useful things before a human remembers to ask.

[ALEX]
Exactly. A hook is a JSON configuration that triggers agent actions automatically at specific events. The key accessibility idea is prevention and timely feedback. Instead of waiting until a late-stage review, the system can intervene on edit, save, commit, pull request creation, or on a schedule.

[JAMIE]
So what lifecycle events are called out?

[ALEX]
The appendix lists editing before save, saving after changes, committing before push, pull request creation before merge, and scheduled runs such as nightly or weekly audits.

[JAMIE]
That maps nicely to the real development rhythm.

[ALEX]
It does. The sample configuration lives in a file like .github slash hooks slash accessibility.hooks.json and contains an array of hook objects.

[JAMIE]
What kinds of hooks are shown in the sample?

[ALEX]
One example is proactive detection on file edit. It watches matching files, routes them to an ARIA specialist, checks a condition such as whether the file includes ARIA-related content, and highlights violations at warning severity. Another example is an edit gate on file save. It watches source files, runs a broader accessibility wizard, and if severity crosses an error threshold, it blocks the save and tells the user to use a fixer agent.

There is also a commit-check hook that runs on commit across all files, uses an accessibility lead agent, and appends scan results to the commit message. And a nightly audit hook runs on a cron schedule, targets documentation Markdown, uses a Markdown accessibility assistant, and creates issues with labels like accessibility and auto-audit.

[JAMIE]
That last one is especially good for repositories that have lots of docs and not enough time for constant manual review.

[ALEX]
Yes. It keeps quality from depending entirely on memory.

[JAMIE]
What properties does a hook configuration typically include?

[ALEX]
The appendix lists a standard set. Name is the unique identifier. Event says when it runs, like on-file-edit or on-commit. Pattern is a glob describing which files it applies to. Agent is the specialist or orchestrator to run. Condition defines whether the hook should trigger. Action defines what happens, such as highlight, block-save, scan-and-report, or create-issue. Severity can be info, warning, or error. Message is custom feedback to the user. Labels support issue creation workflows.

[JAMIE]
And the source also mentions hook examples beyond the sample block.

[ALEX]
Yes, including a proactive ARIA detection example. The full examples in the source are partially truncated in what we received, but the pattern is clear: hooks let you encode accessible quality gates close to the work.

[JAMIE]
There’s also a heading about enabling hooks by platform.

[ALEX]
Right, and this matters administratively. Platform support can vary. Some environments support custom instructions, agents, or automation differently than others. So learners should verify the current support matrix in GitHub documentation, Copilot documentation, changelogs, and custom agent guidance. In other words, the design pattern is stable, but exact enablement depends on the platform’s current capabilities.

[ALEX]
Next comes browser tools for behavioral scanning.

[JAMIE]
Which sounds very different from just reading source files.

[ALEX]
It is. Browser tools let an agent interact with a page as rendered and behaving in the browser. That matters because many accessibility problems show up only when JavaScript runs, focus moves, dynamic content appears, or the layout changes.

[JAMIE]
So what kinds of browser capabilities are implied here?

[ALEX]
The appendix points broadly to available browser tools and using them in agents. In practice, that usually means tools that can open pages, inspect the DOM, interact with controls, follow links, observe focus movement, capture rendered state, and gather page content for analysis.

[JAMIE]
That seems essential for behavioral accessibility issues.

[ALEX]
Yes. Static code analysis can tell you a button lacks an accessible name. Browser automation can go further and reveal whether a modal traps focus incorrectly, whether keyboard order is broken after a menu opens, or whether content injected after a click is actually announced or reachable.

[JAMIE]
The source specifically calls out auditing page accessibility and then doing a smart audit based on page content.

[ALEX]
Right. That means an agent can first inspect what kind of page it’s dealing with, then decide which checks matter most. For example, if it finds a data table, it can prioritize table semantics. If it finds a form-heavy page, it can focus on labels, error handling, and keyboard flow. If it detects media, it can ask about captions or transcripts. That adaptive behavior is often more efficient than running every possible check with equal weight.

[JAMIE]
And what about limitations?

[ALEX]
The appendix explicitly includes browser tool limitations and workarounds. Even when browser tools are powerful, they may have incomplete support for screen reader behavior, authentication flows, highly dynamic apps, or complex stateful interactions. Workarounds often involve combining browser automation with code inspection, logging, targeted prompts, saved test fixtures, or human verification for the final judgment.

[JAMIE]
So learners should hear this clearly: browser-based scans are powerful, but they are not a substitute for all manual testing, especially when real assistive technology behavior matters.

[ALEX]
Exactly right.

[ALEX]
Now let’s talk orchestration, which is where these building blocks become repeatable patterns.

[JAMIE]
The appendix gives named delegation patterns, yes?

[ALEX]
Yes. The first is a parallel scanning pattern described as a web accessibility wizard. The basic idea is that one orchestrator launches multiple specialists at once.

[JAMIE]
That’s the full-audit model we discussed earlier.

[ALEX]
Correct, and the related parallel audit idea is that independent checks can run concurrently to save time and produce broader coverage. This is especially useful when scanning a page or set of pages for multiple issue classes.

[JAMIE]
Then there’s cascading fixes, with a web issue fixer.

[ALEX]
Right. In a cascading fixes pattern, one stage detects issues, another proposes or applies fixes, and then additional specialists verify whether those fixes introduced new problems or failed to resolve the original ones. It’s sequential and corrective, not just diagnostic.

[JAMIE]
So you might detect a missing label, apply a remediation, and then re-run related checks for focus order, naming, or validation feedback.

[ALEX]
Exactly. The third pattern is skill-first workflows, with a testing coach as the example. Here the main agent relies primarily on invoking reusable skills rather than improvising each process from scratch. That produces more predictable test setup, reporting, and teaching behavior.

[JAMIE]
That seems useful for education, because learners can get the same process each time.

[ALEX]
Yes, especially when teaching structured accessibility testing. The fourth pattern is report aggregation, described as a cross-page analyzer. That means collecting findings from multiple pages or scans and synthesizing them into a portfolio view, identifying recurring issues, shared components, or systemic failures.

[JAMIE]
Which is much closer to how teams actually prioritize work. Not just one broken page, but patterns across a whole site or product.

[ALEX]
Exactly.

What comes next?

Continuous evaluation hooks. This extends the hook idea from occasional checks to recurring evaluation. Instead of treating accessibility as a one-time audit, the system keeps measuring over time, perhaps on commits, pull requests, or schedules, so regressions surface early.

[JAMIE]
Then platform-specific skill variants.

[ALEX]
Yes. Not every environment behaves the same. A skill may need variants tuned for GitHub workflows, a particular editor integration, a cloud agent environment, or a local toolchain. Same overall purpose, different implementation details.

That’s an important engineering technique. If scans, lookups, or reference computations are expensive and inputs have not changed, caching results can make agents much faster and cheaper. But the tradeoff is freshness. You need a good strategy for invalidating stale results when files, pages, or dependencies change.

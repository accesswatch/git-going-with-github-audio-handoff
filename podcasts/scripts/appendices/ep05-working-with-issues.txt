Let’s ground this. What is a GitHub issue, really?

[ALEX]
A GitHub issue is a tracked conversation attached to a repository. It has a title, a body, metadata like labels and assignees, a numbered identity such as issue 150, and a timeline of comments and activity.

[JAMIE]
So it’s more structured than a chat message.

[ALEX]
Much more. Issues are durable. They can be searched, filtered, referenced, assigned, grouped into milestones, linked to pull requests, and sometimes broken into sub-issues.

[ALEX]
Sub-issues let you break a larger issue into smaller, trackable child issues. The parent issue can show progress as those child issues are completed.

[JAMIE]
When should someone use sub-issues?

[ALEX]
When a problem is too large to track as one step. For example, “Fix accessibility in welcome dot md” might be split into “Add alt text,” “Fix heading hierarchy,” and “Correct link text.”

[JAMIE]
How do you create one?

[JAMIE]

[ALEX]
Right. And when you read a child issue, GitHub may show its parent relationship too.

[JAMIE]
How is that different from a task list?

[ALEX]
A task list is lighter weight, usually just checkboxes in issue text. Sub-issues are full issues with their own pages, metadata, discussion, and tracking. Use task lists for simple checkboxes; use sub-issues when each piece deserves independent conversation and status.

Let’s shift into maintainer and triager skills. What does managing issues include?

[ALEX]
Closing and reopening issues, assigning them, changing labels, setting milestones, and sometimes transferring or deleting issues if you have the necessary permissions.

[JAMIE]

[ALEX]

[JAMIE]

[ALEX]
Those are more advanced administrative actions. Transfer is for when the issue belongs in another repository. Deletion is rare and usually reserved for spam or mistakes, depending on project policy.

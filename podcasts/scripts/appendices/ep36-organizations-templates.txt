[ALEX]

[ALEX]
Let’s start with your public presence. GitHub has a special profile customization feature that surprises a lot of people. If you create a public repository whose name exactly matches your username, with the same capitalization, the README from that repository is shown right on your GitHub profile.

[JAMIE]
That’s the special profile README.

[ALEX]
Right. It’s basically your introduction page on GitHub. When someone visits your profile, they don’t just see your avatar and a short bio. They can also see the content of that README as regular page content.

[JAMIE]
And that matters because a profile is often the first thing someone checks before deciding whether to follow you, collaborate with you, or trust your contributions.

[ALEX]
Yes. It helps answer simple human questions. Who are you? What are you working on? What are you learning? How can someone reach you?

[JAMIE]
Walk me through creating one.

[ALEX]
Create a new repository. Name it exactly the same as your GitHub username. Make it public. Initialize it with a README. Then edit that README with the information you want visitors to see on your profile.

[JAMIE]
What should go in it?

[ALEX]
A short introduction is a great start. Then maybe your current focus, like technologies you’re learning or projects you’re contributing to. You can include skills, though that part is optional. You can add contact information such as email, LinkedIn, or a personal site. And if you want, a fun fact or two. That last piece can make you feel like a real person instead of a résumé in Markdown.

[JAMIE]
So something like: Hi, I’m Jane Smith. I care about accessibility. I’m learning TypeScript. I contribute to NVDA docs. Here’s how to contact me.

[ALEX]
Exactly. The source gave an example like that, with sections for current focus, skills, contact information, and a fun fact. The bigger lesson is structure. Keep it scannable and honest.

[PAUSE]

[JAMIE]
What makes a good profile README instead of an overwhelming one?

[ALEX]
A few best practices. Keep it concise, because most visitors skim. Update it occasionally so it doesn’t look abandoned. Be authentic rather than stuffing it with buzzwords. Include useful links so people can learn more.

[JAMIE]
And what should people avoid?

[ALEX]
Excessive badges. A wall of fifty technology badges can create visual clutter and a lot of screen reader noise. Also be cautious with auto-generated stats widgets. Those commit charts and flashy profile cards are often inaccessible, sometimes poorly labeled, and not always very meaningful.

[JAMIE]
So a simple profile is better than a noisy one.

[ALEX]
Very often, yes.

[JAMIE]
Let’s make the accessibility part explicit.

[ALEX]
Definitely. Use headings, especially level-two headings, so screen reader users can jump by heading with a key like H. If you include images, provide alt text. Avoid ASCII art, because screen readers read it character by character and it becomes frustrating fast. And test your README with a screen reader before you publish it.

[JAMIE]
What about low-vision considerations?

[ALEX]
Profile READMEs respect GitHub’s dark and light themes, so test in both. Make sure contrast stays readable and that decorative elements don’t become distracting.

[PAUSE]

[JAMIE]
Your profile has more than just the README, right?

[ALEX]
Yes. You can pin up to six repositories, which is a great way to highlight your best work or the projects you want people to notice first. There’s also your contribution graph, which shows activity over the past year. You can’t really customize that, but it does reflect consistent activity like commits, pull requests, and issues. And you can set a temporary status message, like saying you’re away until a certain date.

[JAMIE]
That’s useful for expectation-setting.

[ALEX]
Exactly.

[PAUSE]

[ALEX]
Now let’s shift from self-presentation to supporting others: GitHub Sponsors.

[JAMIE]
This is GitHub’s way of letting people financially support open source maintainers and projects.

[ALEX]
Yes. Think of it as a funding layer for open source work. A developer or project can create a Sponsors profile. Then supporters choose a monthly tier or sometimes a custom amount. The important point in the source is that GitHub sends the money directly to the developer and does not take a cut.

[JAMIE]
Why would someone sponsor a project?

[ALEX]
Three main reasons were highlighted. Sustainability, because many maintainers are volunteers and funding helps them continue. Gratitude, because a tool may have saved you hours of effort. And sometimes sponsor benefits, like priority support, early access, or special community channels.

[JAMIE]
How do you sponsor someone?

[ALEX]
Go to their profile or repository page. Look for the Sponsor button, usually with a heart icon near the top, close to the profile photo area or near the repository controls. Choose a tier or amount, pick a payment method like credit card or PayPal, and complete the process. GitHub sends a receipt, and your sponsorship can appear publicly on your profile if you choose.

[JAMIE]
And for screen reader users?

[ALEX]
The source specifically notes that the Sponsor control is a standard button. On a page, you can move through buttons with the B key in many screen readers until you hear Sponsor.

[JAMIE]
Can regular contributors apply to receive sponsorships too?

[ALEX]
Yes. If you maintain a project or contribute consistently, you can go to the GitHub Sponsors area, join the waitlist or set up sponsors, connect a payout method such as Stripe or a bank account, create sponsorship tiers, and promote that page. The text specifically notes that many accessibility advocates use Sponsors to support work on assistive technology and inclusive design.

[PAUSE]

[JAMIE]
Next up: wikis. These can be a little confusing because some projects use them heavily and others ignore them.

[ALEX]
That’s because a GitHub wiki is one documentation option, not the only one. A wiki is a separate Markdown-based documentation space attached to a repository. It’s useful when you want multi-page docs, tutorials, FAQs, or information that doesn’t belong in the main README.

[JAMIE]
So if a README is getting too long, a wiki might make sense.

[ALEX]
Exactly. Also if you want community-editable documentation. Wikis can be a lightweight place for shared knowledge.

[JAMIE]
When should you not use a wiki?

[ALEX]
If the project already uses GitHub Pages or another docs site, a wiki may be redundant. If you need documentation changes to be version-controlled right alongside code, a wiki may be a poor fit because it is a separate Git repository. And if you want strong customization or tighter review control, a wiki is more limited.

[JAMIE]
How do you access one?

[ALEX]
Go to the repository and open the Wiki tab. If the project has no wiki yet, GitHub may show an option to create the first page.

[JAMIE]
And creating pages?

[ALEX]
Open the Wiki tab, choose New page, add a title and Markdown content, then save. Pages automatically appear in the wiki sidebar for navigation.

[JAMIE]
Anything special for accessibility?

[ALEX]
Yes. The wiki editor is basically the same kind of editor used for issues and pull request comments, so standard Markdown works well. Use headings and lists for structure. Link between wiki pages with double bracket syntax like page title references. And there’s one important caveat: because the wiki is its own Git repository, changes pushed directly to the wiki are not governed by the main repository’s branch protection rules. In practice that means wiki edits may bypass the review process you expect for code. So treat wikis as supplementary, community-editable docs rather than the place for your most critical, tightly controlled documentation.

[PAUSE]

[ALEX]
Now let’s pivot into structural features that shape how repositories are organized: templates, organizations, and settings.

[JAMIE]
This is the part that helps contributors understand how projects are set up behind the scenes.

[ALEX]
Exactly. As a contributor, you won’t always manage these features, but you’ll run into them constantly.

[PAUSE]

[JAMIE]
Let’s start with repository templates. What problem do they solve?

[ALEX]
A template repository is a starter repository. Instead of copying files manually or forking a project when you don’t want shared history, you can create a new repository from the template. It gives you the structure, files, and often default workflows you need to begin.

[JAMIE]
So how is that different from a fork?

[ALEX]
That distinction matters a lot. A fork keeps a relationship to the original repository. It preserves history and signals that your repository is derived from that upstream project, often because you may contribute back. A template gives you a starting snapshot to build your own independent repository. Usually there is no ongoing fork relationship.

[JAMIE]
So if I’m building my own course project from a starter kit, use a template. If I’m planning to contribute changes back to someone else’s codebase, fork.

[ALEX]
Precisely.

[JAMIE]
How does a repository become a template?

[ALEX]
A maintainer enables the template setting in repository settings. Once it’s marked as a template, GitHub can offer a Use this template button.

[JAMIE]
And to use it?

[ALEX]
Open the template repository, choose Use this template, then create a new repository from it. Depending on the interface, you may choose whether to include all branches or just the default branch, though many learners will mainly use the default branch option.

[JAMIE]
Is there an accessibility note here?

[ALEX]
Yes. The source explicitly calls for a screen reader path, so the key practical idea is that the Use this template control is on the repository page and can be reached through standard heading, button, and landmark navigation. For blind users, it’s helpful to navigate to the main repository controls area and listen for Use this template.

[PAUSE]

[JAMIE]
Now organizations. A lot of major open source projects live under organization accounts instead of personal accounts.

[ALEX]
Right. An organization is a shared account structure used by companies, teams, nonprofits, and open source communities. Instead of one person owning everything, the organization owns repositories and manages access through membership and teams.

[JAMIE]
From a contributor’s point of view, why does that matter?

[ALEX]
Because the org structure explains who maintains the project, who has permissions, and how collaboration is grouped. You might be invited to join an organization, or you may just contribute to repositories it owns. Either way, understanding the model helps.

[JAMIE]
How does joining work?

[ALEX]
Usually an owner invites you. You accept the invitation, and then you become a member. Your membership may have certain permissions or team assignments depending on the organization’s setup.

[JAMIE]
And the visibility of that membership can vary, right?

[ALEX]
Yes. Membership visibility can be public or private. Public membership means people can see on your profile that you belong to the organization. Private membership hides that affiliation from public view.

[JAMIE]
That has privacy implications.

[ALEX]
It does. Some people want their affiliations visible for networking or credibility. Others prefer not to list every organization publicly.

[JAMIE]
What about teams inside organizations?

[ALEX]
Teams are subgroups within an organization. They can represent maintainers, documentation writers, accessibility reviewers, release managers, or any other functional grouping. Teams help control access and make notifications more targeted.

[JAMIE]
And if I’m just browsing an organization page, what should I expect?

[ALEX]
You’ll typically find the organization profile, repositories, perhaps pinned projects, member information if visible, and sometimes teams or package info depending on your permissions. For contributors, this page is a helpful map of a project ecosystem.

[PAUSE]

[JAMIE]
Repository settings may sound administrative, but contributors still need to understand them.

[ALEX]
Absolutely. A few settings shape your experience directly. First, repository visibility. A repository can be public, private, or in some cases internal within enterprise contexts. Public means anyone can view it. Private means access is restricted.

[JAMIE]
Second, archived repositories.

[ALEX]
Yes. An archived repository is effectively read-only. You can still inspect the code and issues, but active development has stopped. As a contributor, seeing that a repo is archived tells you not to expect merges or ongoing maintenance.

[JAMIE]
What about repository topics?

[ALEX]
Topics are labels attached to repositories that help with discovery. They might say things like accessibility, screen-reader, docs, python, good-first-issue themes, or framework names. Topics make repositories easier to find via search and topic browsing.

[JAMIE]
And the default branch name?

[ALEX]
That matters because many instructions assume a branch name like main. The default branch is the branch GitHub treats as primary. Pull requests often target it, cloned repositories often check it out first, and some workflows depend on it.

[ALEX]
Now for the broader social layer: stars, watching, following, feeds, discovery tools, lists, and searching for inclusive projects.

[JAMIE]
This is where GitHub starts to feel a bit like a professional social network, but for code and contribution.

[ALEX]
Exactly. GitHub is more than code hosting. It’s a community system.

[PAUSE]

[JAMIE]
Let’s start with stars.

[ALEX]
A star is basically a bookmark plus a public signal of appreciation. When you star a repository, you’re saying this project matters to me, or I want to come back to it.

[JAMIE]
Why does star count matter to projects?

[ALEX]
Stars are an imperfect metric, but they can still indicate visibility, interest, and community recognition. A high star count can attract more contributors or users. But it does not prove quality by itself.

[JAMIE]
How do you star a repo?

[ALEX]
On GitHub.com, open the repository and activate the Star button near the top. Keyboard users can tab to it. Screen reader users can navigate by buttons or landmarks until they hear Star. There may also be a keyboard shortcut in some contexts. And with GitHub CLI, you can star a repository from the command line.

[JAMIE]
The source also expects specific CLI coverage.

[ALEX]
Right. Conceptually, the CLI supports starring a repository, unstarring it, listing your starred repositories, and checking whether you’ve starred a particular repository. Even if learners don’t memorize command syntax here, they should know those actions exist.

[JAMIE]
And viewing your stars on the web?

[ALEX]
That gives you a bookmark list of projects you care about. It becomes more useful over time if you’re intentional.

[PAUSE]

[JAMIE]
How is watching different from starring?

[ALEX]
Watching controls notifications. Starring is mostly bookmarking and signaling. Watching says, keep me informed about activity in this repository.

[JAMIE]
So if I want updates, I watch. If I just want to save it, I star.

[ALEX]
Exactly.

[JAMIE]
What watch levels should people know about?

[ALEX]
There are different notification levels, such as all activity, custom levels in the web interface, releases only, and ignore. The key idea is that watching is granular. You can choose to be highly informed, lightly informed, or explicitly ignore activity.

[JAMIE]
Can all of that be done from the CLI?

[ALEX]
Not equally. The source points out that some granular watch levels, like releases only, require the GitHub web interface and are not available via API alone. The web UI is where you’ll typically choose nuanced watch behavior. The CLI still supports actions like watching a repository for activity, ignoring a repository, listing watched repositories, and stopping watching.

[JAMIE]
That’s a good example of a place where the website still matters.

[ALEX]
Very much so.

[PAUSE]

[JAMIE]
Next is following people.

[ALEX]
Following is about people rather than repositories. When you follow someone, GitHub may surface their public activity in your experience. It helps you build a network around maintainers, educators, accessibility advocates, or contributors whose work you respect.

[JAMIE]
Who should learners consider following?

[ALEX]
People whose work you learn from, projects you contribute to, communities aligned with your goals, and individuals active in accessibility or inclusive open source. Following can become a discovery path.

How do you follow someone?

Visit their profile and activate the Follow button. Screen reader users can use button navigation to find it. The CLI also supports following a user, unfollowing, listing who you follow, listing your followers, checking whether you follow a specific person, and seeing who another user follows.

[JAMIE]
That last one is underrated.

[ALEX]
It really is. Looking at who respected contributors follow can help you discover more of the community.

Then there’s the home feed.

Your home feed is what you see when you log in. It may contain activity from repositories you watch, people you follow, recommendation signals, and other discovery content. Think of it as a personalized activity stream.

[JAMIE]
So it’s not just noise. It can actually help you discover opportunities.

[ALEX]
Exactly. Your feed can show new repositories, releases, issues, or people that matter to your interests. And over time, you can shape it by deciding who to follow and what to watch.

What about Explore?

[ALEX]
GitHub Explore is a discovery area for finding new repositories, topics, collections, and developers. It helps you move beyond the projects you already know.

Any accessibility guidance?

Yes. When using Explore with a screen reader, rely on headings, landmarks, and structured navigation because Explore pages can be visually dense. Move section by section rather than linearly scanning everything.

[JAMIE]
And Trending?

Trending shows what’s popular right now, often with filters by language and time period. There’s also a developers view in some interfaces, so you can discover active people as well as repositories.

Can the CLI show official trending data?

Not directly through an official API endpoint. The source notes that trending repositories are not in the official API, but you can approximate them by searching for recently starred popular repositories, including within a specific language. That’s a useful reminder that CLI and API access sometimes lag behind the full web experience.

Topics are another big discovery tool.

[ALEX]
Definitely. Topics let you browse repositories by category. If you care about accessibility, screen readers, docs, educational resources, or beginner-friendly projects, topics can quickly narrow your search.

[JAMIE]
The source also calls out useful topics for this community.

[ALEX]
Right. Learners in this audience should watch for topics like accessibility, a11y, screen-reader, inclusive-design, documentation, open-source, good-first-issue, and language-specific topics relevant to their stack.

And with the CLI?

[ALEX]
You can search for repositories with a specific topic, and if it’s your own repo, you can add a topic so others can discover it.

GitHub Lists are newer for some learners. What do they do?

[ALEX]
Lists let you organize starred repositories into named collections. So instead of one giant pile of stars, you can group repositories by purpose, such as accessibility tools, learning resources, projects to contribute to, or examples to revisit.

Can you manage lists from the CLI?

[ALEX]
The source says lists are managed through the web interface only. The CLI can still help you view stars, but list management itself is a web feature.

That’s another case where the browser remains important.

Yes.

Absolutely. Search intentionally. Look for repositories focused on accessibility. Search for good first issues in accessibility-related projects. Search for issues labeled both accessibility and help wanted. Combine topic filters and issue filters when possible.

Follow organizations known for accessibility work. Pay attention to who your community follows. If a trusted accessibility advocate follows certain projects or maintainers, that’s often a strong signal.

Meaning?

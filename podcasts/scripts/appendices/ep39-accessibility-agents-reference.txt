[JAMIE]
Now I want the file and scope story, because that’s where people get confused. What applies always? What applies only to some files? What wins in conflicts?

[ALEX]
Great question. The appendix describes scope and priority across multiple levels. The three main scopes are user level, workspace level, and more specific file or folder targeting.

[JAMIE]
User level means personal and cross-workspace?

[ALEX]
Exactly. User-level files are your own preferences and instructions that follow you across projects. Workspace-level files are repository-shared and meant for the whole team. Then there are targeted files, like .instructions.md with applyTo patterns, which narrow behavior to particular file types or paths.

[JAMIE]
And priority?

[ALEX]
Highest priority wins when there’s conflict, and the more specific scope generally overrides the broader one. So a targeted instruction for Markdown files can override general always-on guidance, and repository guidance can override your general expectations where the tool honors that precedence model.

[JAMIE]
What about combining multiple files?

[ALEX]
Multiple files can be merged. The general idea is additive combination when instructions don’t conflict, and priority-based resolution when they do. That’s why being explicit and non-duplicative matters.

[JAMIE]
Can you ground that in locations?

[ALEX]
At user level, think personal, cross-workspace locations. At workspace level, think repo-shared locations inside the project, especially under .github or dedicated customization folders depending on the file type and platform. The appendix emphasizes both categories because discoverability depends heavily on correct placement.

[JAMIE]
What do learners need to know first?

[ALEX]
Three things: location, frontmatter, and applyTo. File locations determine whether the file is discovered. Frontmatter carries metadata. And applyTo is the glob pattern that says which files the instructions should affect.

[JAMIE]
Let’s slow down on applyTo.

[ALEX]
applyTo is how you say, for example, “use these rules for all Markdown files,” or “only for issue template YAML files,” or “only for docs under a certain folder.” It uses glob-style matching. So the power here is precision.

[JAMIE]
And discovery modes?

[ALEX]
That’s about how the environment finds and activates instruction files. Depending on tooling, files may be discovered automatically from expected locations or surfaced through creation workflows in the editor.

[JAMIE]

[JAMIE]
That seems especially good for accessibility. Instead of vague “be accessible” guidance, you can say exactly how Markdown should be structured, or exactly how issue forms should be written.

[ALEX]
Exactly. This reduces drift and improves consistency.

[JAMIE]
And there are practical creation methods.

Learners should know both routes: using the editor’s guided command flow, or just creating the file directly in the proper location with the proper frontmatter.

[JAMIE]
What about writing effective instructions?

[ALEX]
Keep them specific, testable, and scoped. Avoid contradictory guidance. Prefer concrete requirements over aspirational language. And make accessibility requirements explicit, not implied.

How is .prompt.md different?

[ALEX]

How should learners think about skills compared with agents and prompts?

[ALEX]
Skills are reusable capability modules. An agent is a persona plus behavior package. A prompt is a reusable command. A skill is a chunk of specialized know-how that can be loaded when needed.

[JAMIE]

Before we wrap up, can we do a compact decision guide? If I need to customize behavior, how do I choose the right primitive?

[ALEX]
Absolutely.
If you want broad, always-on project guidance, use .github/copilot-instructions.md.
If you want tool-agnostic or monorepo guidance, consider AGENTS.md, with caution around nested behavior.
If you want Claude Code compatibility, use CLAUDE.md where appropriate.
If you want rules only for certain files, use .instructions.md with applyTo globs.
If you want a named assistant persona with defined tools and behavior, create a .agent.md file.
If you want a reusable slash command, create a .prompt.md file.
If you want reusable specialized know-how, define a skill with SKILL.md.
If you want lifecycle automation or policy enforcement, use hooks JSON.
If you want your own personal defaults and communication preferences, use preferences.md.

[ALEX]

[JAMIE]

[ALEX]
Then forcing it to on is often the right move.

[JAMIE]
What else matters right away?

[ALEX]
Editor dot accessibilityPageSize controls how many lines are read with page navigation in screen reader mode. Default is ten. That sounds minor, but it changes how much content you hear at a time.

[JAMIE]
So it’s part comfort, part efficiency.

[ALEX]
Exactly. Then there are several visual guide settings, like bracket pair guides, horizontal bracket guides, active bracket highlighting, indentation highlighting, and indentation guides. [JAMIE]
So if I’m blind and not using that visual information, I can simplify.

[ALEX]
Yes. Similarly, editor dot minimap dot enabled is visual only, and the recommendation is to turn it off for screen reader users. Editor dot renderWhitespace is also important. The recommended value for screen readers is none, because hearing code is hard enough without trying to manage a UI designed to show whitespace visually.

[JAMIE]
What about line wrapping?

[ALEX]
Editor dot wordWrap is recommended on for screen reader users, especially with long lines. It can reduce horizontal navigation friction. And workbench dot editor dot enablePreview is worth considering too. If preview mode stays on, opening files can reuse one tab. Turning it off means files open in persistent tabs, which many users find easier to track.

[JAMIE]

[ALEX]
Editor dot hover dot enabled can stay on, but the key point is this: for accessible reading of hover content, use Accessible View with Alt+F2.

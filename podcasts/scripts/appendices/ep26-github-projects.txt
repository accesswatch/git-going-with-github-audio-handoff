And before we get into the how, what changed with the newer Projects system?

[ALEX]
A lot. GitHub had an older generation sometimes called classic projects. That older system was mostly Kanban only. The newer Projects version gives you three layouts: Board, Table, and Roadmap.

[JAMIE]
So not just sticky notes in columns anymore.

[ALEX]
Right. You also get custom fields now, which is a huge shift. Instead of just title and labels and assignees, you can add your own fields like text notes, numeric estimates, dates, single-select choices like priority, and iteration fields for sprint-style planning.

What about automation?

[ALEX]
Also improved. Classic projects had very limited built-in behavior. Projects v2 supports built-in workflows and integrates with GitHub Actions. Filtering is much stronger too. You can query by labels, assignees, statuses, and custom field values. Cross-repository tracking is supported, and the API story is more powerful through GraphQL.

[JAMIE]
And if someone sees old references to Projects Beta?

[ALEX]
That was the earlier rollout period for this newer system. Today, Projects v2 is the standard one people mean.

[JAMIE]
Okay, I want to create one. Where can a project live?

[ALEX]
Two common places. At the organization level, which makes sense when a team wants shared planning across multiple repositories. Or at the repository level, which is narrower and centered on one repo.

[JAMIE]
Let’s start with organization-level creation.

[ALEX]
You navigate to the organization page, open the Projects tab, activate the New project button, choose a template or start blank, give the project a name, and activate Create project.

[JAMIE]
What kinds of templates are we talking about?

[ALEX]
Things like a blank table, blank board, or a more opinionated setup like a feature release template. Templates can save time if you already know the workflow pattern you want.

[JAMIE]
And from a repository?

[ALEX]
Very similar. Go to the repository, open its Projects tab, then use Link a project or New project, depending on what GitHub shows there. If you create a new one, the template and naming flow is basically the same.

[JAMIE]
Can we pause on the screen reader flow? Because this is one of those spots where sighted walkthroughs can hide important interaction details.

[ALEX]
Absolutely. A typical screen reader path is: go to the org or repo Projects tab, tab to the green New project button, press Enter. The template dialog opens. Use arrow keys to move among template cards, each announcing its name and description. Press Enter on the one you want. Then type the project name in the name field, tab to Create project, press Enter, and you’ll land in the new empty project.

[JAMIE]
And after creation, what should users expect?

[ALEX]
Usually an empty project in Table view. There’s often a plus Add item control near the bottom. For screen reader users, tabbing to that control is a reliable way to start adding work. For low-vision users, one practical note: at high zoom, that add-item row may be off-screen, so scrolling down matters.

[JAMIE]
And visually, the New project button tends to be easy to spot?

[ALEX]
Yes, typically a green high-contrast button on the Projects tab.

[PAUSE]

[JAMIE]
Now let’s talk about the three layouts, because this seems like the heart of the system.

[ALEX]
It is. The default is usually Table view. Think spreadsheet. Each row is an issue, a pull request, or a draft item. Each column is a field: title, status, assignee, labels, priority, dates, repository, or any custom fields you create.

[JAMIE]
What is Table view best at?

[ALEX]
Bulk editing, sorting, filtering, and scanning lots of structured information quickly. If you want the most information density, this is usually the winner.

[JAMIE]
And accessibility-wise?

[ALEX]
It’s generally the most accessible layout. Keyboard movement is strong. Common shortcuts and navigation patterns include jumping to the table, moving up and down between rows, left and right between columns, pressing Enter to open the item detail panel in the right sidebar, Escape to close that panel, Space on a cell to edit the value, and F2 on the title cell for inline title editing.

[JAMIE]
What does a row sound like in a screen reader?

[ALEX]
Something like: title first, then field-value pairs. For example, “Fix keyboard trap in modal dialog, Status: In Progress, Assignee: alice, Priority: High, Labels: bug, accessibility.” That row-style verbal summary is one reason Table view works so well nonvisually.

[JAMIE]
So if someone is blind and choosing a default working view, Table is probably the safest choice.

[ALEX]
Yes. Table first, especially for editing.

[PAUSE]

[JAMIE]
What about Board view?

[ALEX]
Board view is the Kanban-style layout. Columns represent status values, and cards sit inside those columns. It’s excellent for workflow tracking, work-in-progress conversations, and quick standup-style reviews.

[JAMIE]
But I know there’s an accessibility catch here.

[ALEX]
There is. Drag-and-drop is not keyboard accessible in the way many users need. So the accessible method for changing a card’s column is not dragging. Instead, open the card, find the Status field in the right-side detail panel, activate the status dropdown, choose the new value, and then close the panel.

[JAMIE]
That’s such an important pattern: don’t imitate the visual gesture if the underlying field can be edited directly.

[ALEX]
Exactly. The column is just a visual expression of the Status field. Change the field, and the card moves.

[JAMIE]
Keyboard-wise, how do people move around the board?

[ALEX]
Tab commonly moves among columns and cards. Enter on a card opens the detail panel. In some contexts, heading or region navigation can help jump among board areas. And within a card context, Space may toggle selection for bulk actions.

[JAMIE]
For low-vision users, any practical warning?

[ALEX]
Yes. Board columns can overflow horizontally, so at high zoom you may need to scroll sideways. Card details like label badges or avatars can become small and crowded, so zooming in on a single card or switching to Table view can be easier.

[PAUSE]

[JAMIE]
And Roadmap view?

[ALEX]
Roadmap is a horizontal timeline. Items appear as bars spanning dates, often start dates and due dates, or iteration windows. It’s useful for release planning, sprint capacity planning, and communicating timing to stakeholders.

[JAMIE]
This sounds visually rich, but maybe less accessible directly.

[ALEX]
That’s the right instinct. Roadmap is chart-like, so for screen reader users it’s best treated as a visual summary rather than the primary editing surface. The underlying data is still accessible in Table view. So the recommended workflow is: edit and verify in Table, then switch to Roadmap if you want the big timeline picture.

[JAMIE]
How do items get onto the roadmap timeline?

[ALEX]
They need date information. You can add Date custom fields like Start Date and Due Date, or sometimes use milestone-related timing depending on your setup. Items without dates usually sit outside the timeline grouping.

[JAMIE]
And low-vision users may need more zoom because those bars can be narrow.

[ALEX]
Exactly.

[PAUSE]

[JAMIE]
Let’s move into custom fields, because this is where Projects starts feeling like a real planning database.

[ALEX]
That’s a great way to say it. Projects includes default fields like Title, Assignee, Labels, Milestone, and Repository, but its real power comes from adding your own structured fields.

[JAMIE]
Walk us through the major field types.

[ALEX]
Text fields are free-form notes, useful for links, criteria, or reminders. Number fields are for numeric values like estimates or story points. Date fields store deadlines or start dates. Single select gives you a dropdown of predefined choices, great for priority, size, risk, or type. And Iteration fields support sprint or cycle planning.

[JAMIE]
How do you create one?

[ALEX]
From Table view, you usually move to the far end of the header row and activate the plus button that adds a field. Then you choose the field type, provide the field name, and configure options if needed. For single-select, for example, you define the allowed values such as P0, P1, and P2.

[JAMIE]
What about the screen reader path for that?

[ALEX]
The important pieces are: get into Table view, move across the header row until you reach the add-field control, activate it, select the field type from the menu or dialog, then fill in the naming and option fields. Because this interaction can involve horizontal movement across many columns, it may take some patience, especially in a busy project.

[JAMIE]
And once the field exists, how do you edit a value on an item?

[ALEX]
Usually two ways. In Table view, navigate to the cell and activate it, often with Space, then choose or type the value. Or open the item detail panel and edit the field there in the sidebar. The panel method is often clearer for screen reader users because it exposes labeled controls in a more linear form.

[JAMIE]
So if a learner finds the grid editing model awkward, the sidebar is a good fallback.

[ALEX]
Very much so.

[PAUSE]

[JAMIE]
Now let’s add work items. What exactly can live inside a project?

[ALEX]
Three main categories: existing issues, existing pull requests, and draft issues. Drafts are especially handy when you want to capture work before a formal issue exists in a repository.

[JAMIE]
How do you add an existing issue or pull request?

[ALEX]
Use the add-item control in the project, then search for the issue or PR you want. Depending on the project’s scope and permissions, you can pull in items from linked repositories or other repos in a cross-repository setup.

[JAMIE]
And a draft issue?

[ALEX]
A draft issue is created directly in the project without immediately creating a repository issue. It’s useful for planning, brainstorming, backlog triage, or work that isn’t ready to be assigned to a repo yet.

[JAMIE]
Then later you can promote it?

[ALEX]
Yes. Promoting a draft turns it into a real GitHub issue in a repository. That’s a nice bridge between planning and execution.

[JAMIE]
So a team can start fuzzy, then formalize later.

[ALEX]
Exactly.

[JAMIE]
What about bulk editing?

[ALEX]
That’s one of Table view’s strengths. You can select multiple items and update shared properties together, such as status, iteration, assignee, or other field values. In practice, this is useful when moving a batch of tasks into a sprint, closing out a release phase, or assigning a common label or priority pattern.

[JAMIE]
Accessibility tip there?

[ALEX]
Know how selection is announced, and confirm the set of selected items before applying the bulk change. In any bulk operation, the risk isn’t understanding one item; it’s accidentally changing many.

[JAMIE]
Let’s get to automations. What can Projects do on its own?

[ALEX]
GitHub Projects includes built-in workflows and can also work with GitHub Actions. The idea is to reduce repetitive project maintenance.

[JAMIE]
Where do people access automations?

[ALEX]
Typically through project settings or a workflows area associated with the project. The exact UI can shift over time, but conceptually you’re looking for automation or workflow configuration within the project’s controls.

[JAMIE]
And what kinds of built-in workflows are we talking about?

[ALEX]
Common examples include automatically adding items to a project when they meet certain conditions, or updating fields based on status changes and project events. The source emphasizes native workflows and GitHub Actions integration rather than treating the project as a totally manual tool.

[JAMIE]
I noticed auto-add was specifically called out.

[ALEX]
Yes. Auto-add is especially useful when you want matching issues or pull requests to appear in a project automatically. For example, perhaps all issues with a certain label, or work from a specific repository, should flow into a tracking project without someone manually adding them every time.

[JAMIE]
That sounds powerful, but also like something you’d want to test carefully.

[ALEX]
Absolutely. Start with a narrow rule, verify the results, then expand. Automation should reduce work, not create surprise clutter.

[JAMIE]
And GitHub Actions integration means you can go beyond built-in toggles?

[ALEX]
Right. Actions can update project fields, move work through states, or synchronize project data with repository events in more advanced workflows. If your team already uses Actions, Projects becomes another place where automation can keep planning data current.

[PAUSE]

[JAMIE]
Now the sprint story: iterations.

[ALEX]
Iterations are GitHub’s sprint-like field type. Instead of storing just a random date, an iteration field defines repeating cycles, such as two-week sprints.

[JAMIE]
How do you create one?

[ALEX]
Create a custom field and choose Iteration as the type. Then configure the cadence, such as duration and start date. GitHub generates the repeating iteration values from that setup.

[JAMIE]
And using iterations means assigning each item to one of those sprint buckets?

[ALEX]
Exactly. Once the field exists, items can be assigned to a specific iteration, making it easy to filter by current sprint, next sprint, or backlog. This is often clearer than manually typing sprint names in a text field because the values are structured and date-aware.

[JAMIE]
What are iteration insights?

[ALEX]
They’re the planning benefits you get from that structure: seeing which items are in the current cycle, comparing upcoming cycles, and using roadmap or filtered views to understand sprint load and timing. In other words, the insight comes from consistent structured data.

[JAMIE]
So it’s not just labeling work “Sprint 4.” It’s giving GitHub enough semantics to reason about time.

[ALEX]
Perfectly put.

[JAMIE]
Views and filters feel like another big productivity layer.

[ALEX]
They are. A project can have multiple saved views, each showing the same underlying items differently.

[JAMIE]
So one project, many perspectives.

[ALEX]
Exactly. You might have a Table view for backlog grooming, a Board view for daily status, and a Roadmap view for release timing. You can also create specialized views like “My open work,” “Current sprint,” or “Only accessibility bugs.”

[JAMIE]
How do you create a view?

[ALEX]
You usually start from the current layout, apply filters, grouping, sorting, and visible fields the way you want, then save that configuration as a named view.

[JAMIE]
Let’s unpack filter syntax a bit.

[ALEX]
Projects supports query-style filtering using fields and common GitHub metadata. The exact syntax can include things like status values, labels, assignees, repository names, and custom fields. The important conceptual point is that filtering is expressive. You are not limited to one simple dropdown; you can build meaningful queries.

[JAMIE]
So for example, I might isolate items assigned to me, with a certain label, in a particular status.

[ALEX]
Exactly. Or only items from one repository inside a cross-repo project. Or only the current iteration. Filters become much more valuable once your project has structured fields.

[JAMIE]
And grouping?

[ALEX]
Grouping changes how items are clustered visually. In Table view, you might group by Status, Assignee, Priority, or Iteration. That lets you treat one project almost like several logical sections without creating separate projects.

[JAMIE]
Sorting is more straightforward, I assume.

[ALEX]
Yes. Sorting orders items by one or more fields. Useful examples include due date ascending, priority descending, or recently updated work first. Combined with grouping, sorting can make a large project much easier to read.

Cross-repository projects feel especially important for open source maintainers and org-level planning.

[ALEX]
Definitely. One of the biggest advantages of Projects v2 is that it can track work from multiple repositories together.

[JAMIE]
How do you add a repository to a project?

[ALEX]
At a high level, you configure the project so it can include items from that repository, then add or auto-add issues and PRs from it. The exact UI wording can vary, but the concept is: this project is not limited to one repo. It can aggregate work across several.

[JAMIE]
And then the Repository field becomes important.

[ALEX]
Very. Once you’re mixing repos, the Repository field tells you where each item came from. It’s useful in Table view for filtering, sorting, and basic orientation. If two repositories both have an issue named something vague like “Update docs,” the Repository field keeps those items distinguishable.

[JAMIE]
So cross-repo planning without the Repository field visible can get confusing fast.

[ALEX]
Exactly. In a multi-repo view, showing repository is often a best practice.

Once you’re inside a project, what’s the general model?

[ALEX]

[JAMIE]

[ALEX]
Yes. Region awareness matters. In general, it helps to locate the current view, then the filter bar, then the content area. When you open an item, understand that focus may move into a right sidebar panel with editable fields.

[JAMIE]
Give us the Table view mental map.

[ALEX]
Header row of fields across the top. Rows of items below. Arrow navigation can move within the grid. Enter opens details. Space may edit a cell. Because it’s grid-like, listen carefully for row and column context so you always know which field you’re on.

And Board view?

[ALEX]

[JAMIE]

[ALEX]
This right sidebar appears from any view after opening an item. It’s often the most linear and therefore accessible place to review and edit title, status, assignees, labels, custom fields, and other metadata. Escape commonly closes it.

[JAMIE]

[ALEX]
The filter bar is where you type or refine project queries. For screen reader users, that’s a powerful control because it can quickly reduce noise. If a project feels overwhelming, filtering down to one assignee, one status, one repo, or one iteration can make the experience much more manageable.

[ALEX]
Now let's move to Dependabot. Dependabot is GitHub's automated system for monitoring dependencies and helping keep them updated.

[JAMIE]
And it really does two related but distinct jobs.

[ALEX]
Exactly. First, Dependabot alerts tell you when a package your project depends on has a known vulnerability. Second, Dependabot can open pull requests to fix those problems, and it can also open routine version update PRs even when the change isn't specifically security-related.

[JAMIE]
Let's unpack the alerts first. What triggers one?

[ALEX]
If a vulnerability is discovered in a dependency used by your project, GitHub can create a Dependabot alert. You'll usually see details like the package name, ecosystem such as npm, the vulnerable version range, the fixed version, an advisory identifier such as a GHSA code, and a short description of the risk.

[JAMIE]
So the alert is not just saying, "Something is bad." It gives actionable specifics.

[ALEX]
Exactly. And severity matters. The source laid out the common levels using CVSS ranges. Critical, roughly 9.0 to 10.0, means very serious and likely needing immediate attention. High, around 7.0 to 8.9, is significant risk and usually should be fixed within the current sprint or very soon. Medium, 4.0 to 6.9, still matters but may require specific conditions to exploit. Low, 0.1 to 3.9, is lower urgency but should still be addressed when practical.

[JAMIE]
That's helpful, because learners often see severity labels without understanding the difference between "drop everything" and "plan this responsibly."

[ALEX]
Right. And when you see a dependency alert as a contributor, a good process is: first, read the advisory so you understand scope. Second, check whether the project actually uses the vulnerable code path. Third, the fix is usually to update the dependency to the patched version. Fourth, look in Pull Requests, because Dependabot Security Updates may already have opened a PR for you.

[JAMIE]
I like that middle step. Sometimes the vulnerable package is present, but the exact dangerous feature isn't used.

[ALEX]
Yes, though you still generally want to patch it. Understanding actual exposure helps prioritize and communicate clearly.

[JAMIE]
What does a Dependabot security update PR typically look like?

[ALEX]

[JAMIE]

[ALEX]

[ALEX]

In plain language, that means GitHub checks for dependency updates each week and opens at most a set number of update pull requests at a time.

[ALEX]

[ALEX]
Next is secret scanning, and this one is urgent by nature. Secret scanning looks for accidentally committed credentials such as API keys, tokens, and passwords.

[JAMIE]
This is the one that can stop your push before it lands, right?

[ALEX]
Yes. Push Protection is the real-time prevention layer. If GitHub detects a known secret pattern during a git push, it can block the push and return an error. The example in the source showed a GH013 repository rule violation, followed by lines identifying the commit, file, secret type, and line location.

[JAMIE]
Which is actually pretty useful if you slow down and read it.

[ALEX]
Exactly. If your push is blocked, do three things. First, remove the secret from the file immediately. Second, rotate the exposed credential. GitHub may automatically revoke detected GitHub tokens, but you should still treat the credential as compromised. Third, if you suspect a false positive, a bypass may be possible with justification, but only after careful investigation.

[JAMIE]
And the best practice is to avoid hardcoding secrets in the first place.

[ALEX]
Always. The source gave a very clear bad-versus-good example. Bad: assigning a token string directly in code. Good: reading it from an environment variable like process dot env dot GITHUB underscore TOKEN.

[JAMIE]
That distinction matters for accessibility too, because if you're reviewing code by speech or braille, variable names are easier to reason about than a long credential string buried in a file.

[ALEX]
Well said. And for public repositories, GitHub can also scan existing commits and create secret scanning alerts for already-exposed secrets. Those alerts appear under Security and quality, then Secret scanning.

[JAMIE]
What's the right response if you find one?

[ALEX]
Treat it as urgent. Especially if it's a token or API key, assume the credential may have been exposed for a long time. The immediate priorities are revocation or rotation, removing the secret from the codebase, and following project procedures for cleanup and disclosure.

[JAMIE]
Let's do the accessibility layer here too.

[ALEX]

For sighted users, visual banners and icons can help, but the key lesson is the same for everyone: never hardcode tokens.

[ALEX]
Now to code scanning and CodeQL. This is different from dependency scanning. Dependabot is about packages you use. Code scanning is about vulnerabilities in your actual code.

[JAMIE]
So this is static analysis.

[ALEX]
Exactly. GitHub's built-in system commonly uses CodeQL, which analyzes code patterns and logic. The source explicitly mentioned risks like SQL injection and cross-site scripting, and this category also generally includes other dangerous data flow or unsafe coding patterns.

[JAMIE]
How should a learner think about a code scanning alert when they open one?

[ALEX]
Read it as a structured explanation. A code scanning alert usually tells you what type of vulnerability was found, where it appears in the code, how severe it is, and often a path of reasoning from source to sink, meaning how untrusted input could reach a dangerous operation.

[JAMIE]
And that's the part people sometimes find intimidating.

[ALEX]
Yes, but you can break it down. Ask: what is the risky behavior, where in the file is it, what data is involved, and what safer pattern should replace it? For example, if the issue is SQL injection, the fix may involve parameterized queries rather than string concatenation. If it's XSS, the remedy may involve proper escaping, sanitization, or avoiding unsafe rendering patterns.

Next is private vulnerability reporting, which is a major open source professionalism topic.

[JAMIE]
Meaning: if you find a serious bug, do not immediately open a public issue describing exactly how to exploit it.

[ALEX]
Exactly. If the repository has private vulnerability reporting enabled, you should use that pathway to submit the report confidentially.

How does that usually work from the contributor side?

[ALEX]

The exact timing and method depend on context, but the key is to act ethically and document your attempts to contact the maintainers.

[JAMIE]
Let's build the vocabulary, because this topic gets confusing fast if all the words blur together.

[ALEX]
Good idea. A workflow is the complete automated process described in one YAML file. A repository can have many workflows.

A job is one major unit inside a workflow. Jobs usually run on their own machine environment.

A step is one task inside a job, like running a shell command or calling a reusable action.

An action is a reusable unit of automation, almost like a plugin. Many public actions are shared through GitHub Marketplace.

A runner is the machine that executes the job. Often it's a GitHub-hosted virtual server.

A trigger, or event, is what starts the workflow. Push, pull request, schedule, and manual launch are common examples.

A status check is the pass or fail result that appears on a pull request.

An artifact is a file produced by a workflow, like a report, build output, or downloadable package.

A secret is an encrypted value stored in repository settings so workflows can use sensitive data without exposing it directly.

And YAML is the format used to write workflow files. Indentation matters a lot.

[JAMIE]
There are also two little YAML keys that people see constantly: on and runs-on.

[ALEX]
Exactly. The key called on defines the trigger. The key called runs-on tells GitHub what operating system or runner environment to use, like ubuntu-latest, windows-latest, or macos-latest.

[JAMIE]
Let's tackle the emotional side and the practical side together: what should you do when a check fails?

[ALEX]
Step one: don't panic. A failed check is information, not a judgment about your worth or your ability.

Step two: navigate to the failed check. From the PR checks section or the Actions tab, open the failed run.

Step three: find the failing step. Don't stop at the workflow name. You need the exact job and exact step that failed.

Step four: read the error message carefully. Look for the command that ran, the file involved, the line number if one is given, and whether the issue is a test failure, formatting issue, missing dependency, syntax error, or something else.

Step five: fix the issue locally or in the web editor, depending on the kind of change and the repository's workflow. Some problems are quick text edits. Others are best fixed in VS Code where you can run tools locally.

Step six: commit and push the fix. That usually retriggers the relevant checks automatically.

Step seven: if you're stuck, ask for help. Leave a comment on the PR or issue, explain what check failed, what you already tried, and what part you don't understand.

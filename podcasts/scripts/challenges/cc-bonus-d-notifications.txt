[ALEX]
Now let’s step back and answer the basic question: what generates a notification?

[JAMIE]
Because once you understand the causes, the inbox makes more sense.

[ALEX]
You get notifications when someone @mentions your username in an issue, pull request, or discussion. You get them when a pull request is assigned to you for review. You get them when an issue or PR is assigned to you. You also get them when there’s activity on a thread you’re subscribed to, which can happen because you commented, were mentioned, or chose to subscribe. A CI or Actions failure on your PR can generate a notification. And if you are watching a repository broadly enough, a published release can also notify you.

[JAMIE]
So GitHub is combining direct involvement, explicit assignment, and repository-level subscriptions.

[ALEX]
Exactly. And that is why notification management matters. If you subscribe too broadly, the signal gets buried.

[PAUSE]

[JAMIE]
Let’s unpack subscription levels, because this is where many learners accidentally create chaos.

[ALEX]
For each repository, you choose how much you want to hear.
Participating and @mentions means only conversations where you participated, were assigned, or were mentioned.
All Activity means every issue, every comment, every pull request, every update.
Ignore means no notifications from that repository at all.
Custom gives you finer control, such as issues only, pull requests only, or releases.

[JAMIE]
And the recommendation for most repositories in this workshop is Participating and @mentions.

[ALEX]
Yes. It keeps you in the loop without flooding you.

Now we should discuss account-level notification settings, because repository watch settings are only part of the picture.

[JAMIE]

[ALEX]
Yes. At github dot com slash settings slash notifications, you can configure how GitHub contacts you. The reference setup contrasts defaults with a more intentional configuration.

[JAMIE]
Start with the default problem.

[ALEX]
With default settings, GitHub may send email notifications for every issue and pull request in every repository you watch, every comment on threads you’ve participated in, and every CI status update. That becomes overwhelming fast. Many people end up ignoring all GitHub email, which defeats the purpose.

[JAMIE]
So the reference solution is not saying all defaults are bad. It’s saying defaults often create too much volume.

[ALEX]
Exactly.

[PAUSE]

[ALEX]
A more productive setup looks like this.
For Participating notifications, email on and web on. If someone replies where you’re actively involved, you usually want to know.
For Watching notifications, email off and web on. That means watched repository activity is available in GitHub when you choose to check it, but it does not constantly interrupt your email.
For GitHub Actions, email off for successful runs and on for failed runs only.

[JAMIE]
That last one is elegant. A passing check usually doesn’t require action.

[ALEX]
Exactly. A green check on the PR page is enough. Failures are what need attention.

[JAMIE]
And repository-level watching?

[ALEX]
For repositories where you actively contribute, watching all activity can make sense. For repositories you read occasionally, use Custom, often issues and pull requests only. For repositories you’ve finished with, unwatch them.

[JAMIE]
That last one may be the most life-changing.

[ALEX]
It often is. Old subscriptions quietly clog your feed.

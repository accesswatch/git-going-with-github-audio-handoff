[JAMIE]
Okay. So now let’s connect culture to workflow. Where do all these conversations actually happen?

[ALEX]
Mostly around pull requests, using GitHub Flow. GitHub Flow is the lightweight contribution model this workshop uses. The six steps are straightforward: create a branch off main, make your changes and commit them, open a pull request, discuss and review, pass status checks, and then merge.

[JAMIE]
That sounds simple, but each step carries social meaning.

[ALEX]
Very much so. A branch says, “I’m working on one thing.” A pull request says, “Here’s the change, here’s why, please review it.” Review discussion turns code into conversation. Status checks protect the whole project. And merge means the work is accepted into shared history.

[JAMIE]
Why does this model work so well for open source?

[ALEX]
Because main stays deployable, branches are cheap, pull requests centralize discussion, and small focused changes move faster. It also supports accessibility-friendly collaboration because the unit of work is explicit and reviewable.

[JAMIE]
And this is different from Git Flow.

[ALEX]
Yes. Git Flow is older and more complex. It uses long-lived branches like `main` and `develop`, plus temporary `feature/`, `release/`, and `hotfix/` branches. It fits teams with scheduled releases, versioned software, or enterprise processes. GitHub Flow is lighter: usually just `main` plus short-lived feature branches. For a workshop, web projects, and open source contributions, GitHub Flow is easier to teach, easier to review, and easier to recover from.

[JAMIE]
So if I see `develop` and `release/1.2.0`, I’m probably in Git Flow territory.

[ALEX]
Exactly. But for this workshop, think one branch per task, PR back to main, and keep main healthy.

[JAMIE]
That ties into the unwritten rule: one thing per branch.

[ALEX]
Yes. One branch should represent one coherent change. That helps you make atomic commits, write focused PR descriptions, and receive useful reviews. If you mix a README rewrite, a bug fix, and a formatting pass in one branch, the reviewer has to untangle three stories at once.

[PAUSE]

[JAMIE]
What about keeping your fork current? That seems administrative, but it affects collaboration.

[ALEX]
It absolutely does. If your fork is stale, you create avoidable conflicts and review noise. So you should sync your fork, especially before starting new work, after the upstream project has moved, or after your last contribution was merged long ago.

[JAMIE]
And there are multiple ways to do that.

[ALEX]
Three common ones. Easiest is the GitHub web interface using the Sync fork option. Another is the command line in the VS Code terminal. That requires a one-time setup: add the upstream remote. Then the sync process is four steps. Switch to your main branch. Fetch upstream changes. Merge upstream main into your local main. Push the updated main back to your fork on GitHub.

[JAMIE]
So in plain language: get onto your own main, bring down the upstream changes, combine them, and publish the updated result to your fork.

[ALEX]
Exactly. And there’s also GitHub Desktop for people who prefer a GUI. The important concept is not the tool but the habit: sync before branching so you start from current code.

[PAUSE]

[JAMIE]
Let’s talk commits, because group work gets messy fast if commit history is muddy.

[ALEX]
Very true. Good commit messages help reviewers and future maintainers understand what happened. The usual format is a short required first line that states the change clearly, then an optional body explaining why, and an optional footer for issue references or metadata.

[JAMIE]
What makes the first line good?

[ALEX]
It should be concise and specific. Think “Add alt text to homepage feature cards” rather than “stuff” or “updates.” The body can explain reasoning, tradeoffs, or context. The footer might include something like `Closes #42` if the project uses issue linking.

[JAMIE]
And atomic commits?

[ALEX]
An atomic commit contains one coherent, complete change. That makes review easier, reverts safer, and collaboration calmer. Common mistakes include vague messages, giant mixed-purpose commits, or committing generated clutter with meaningful edits.

[JAMIE]
So the same principle keeps showing up: one branch, one PR, one clear idea per commit when possible.

[ALEX]
Exactly.

[PAUSE]

[JAMIE]
Now let’s get into the human side. You said open source communication happens in writing, asynchronously, and in public. Can we unpack that?

[ALEX]
Of course. In writing means people cannot hear your tone of voice, so clarity matters. Asynchronously means the other person may reply hours later, so your message has to stand on its own. In public means your comment may help many future readers, but it also means bluntness can feel amplified.

[JAMIE]
So that’s why feedback needs structure.

[ALEX]
Yes. Helpful feedback usually does five things. First, acknowledge what’s working. Second, identify the specific concern. Third, explain why it matters. Fourth, suggest a path forward when you can. Fifth, signal the weight of the concern.

[JAMIE]
That last one is subtle.

[ALEX]
And very useful. Not every comment is blocking. Sometimes it’s a required fix. Sometimes it’s a suggestion. Sometimes it’s a question. Being explicit reduces stress.

[JAMIE]
How should the language sound?

[ALEX]
Prefer “we” language or talk about the code, not the person. Use tentative language when you’re uncertain. Remember contributors may come from different cultures and may be working in a second language. Avoid urgency markers unless something is genuinely urgent.

[JAMIE]
So instead of “You did this wrong,” maybe “This heading order may confuse screen reader navigation; could we move this H3 under an H2?”

[ALEX]
Perfect. That addresses the artifact, explains the impact, and proposes a fix without attacking the author.

[PAUSE]

[JAMIE]
What about commenting etiquette more generally?

[ALEX]
Keep comments focused. Don’t bundle five unrelated concerns into one comment if separate comments would be clearer. Don’t leave comments unresolved forever; if the issue is addressed, resolve the conversation. Don’t pile on by repeating someone else’s point unless you’re adding necessary context. Reactions can be useful lightweight signals of agreement or thanks.

[JAMIE]
And saved replies are a big accessibility win, right?

[ALEX]
They can be. Saved replies let you store reusable comment templates for common situations. Useful for thanking contributors, asking for tests, clarifying reproduction steps, or giving a standard accessibility explanation. On GitHub, you can create a saved reply in your settings, then insert it while writing a comment. For screen reader users, this can reduce repetitive typing and cognitive load, especially for phrases you use often.

[JAMIE]
So a saved reply might be something like, “Thanks for this contribution. I noticed a heading hierarchy issue that may affect screen reader navigation. Could we adjust the heading levels so they don’t skip?”

[ALEX]
Exactly. Then you customize the details for the current PR.

[PAUSE]

[JAMIE]
Let’s split etiquette by role: reviewer versus author.

[ALEX]
For reviewers: review the code, not the person. Don’t gatekeep knowledge by implying someone should already know everything. Ask questions instead of making demands when appropriate. Distinguish opinion from requirement. And approve explicitly when the work is ready.

[JAMIE]
That “distinguish opinion from requirement” part seems huge.

[ALEX]
It is. “This must be fixed because it breaks keyboard navigation” is different from “I prefer a shorter sentence here.” If you blur those, authors waste effort guessing what is essential.

[JAMIE]
And for authors?

[ALEX]
Say thank you. Don’t take feedback personally. Explain your choices when context matters. Surface blockers early instead of silently stalling.

[JAMIE]
What about accessibility-specific comments?

[ALEX]
Inclusive commenting matters a great deal. Accessibility issues are real product issues, not optional polish. But the tone should stay respectful and educational. Explain impact: for example, missing alt text leaves image meaning unavailable to some users. Avoid framing accessibility as a burden or special interest request.

[JAMIE]
And there’s also that “good first issue” social contract.

[ALEX]
Yes. If a project labels something “good first issue,” it should be genuinely approachable, reasonably scoped, and documented enough that a newcomer has a fair chance. That’s part of community trust.

[PAUSE]

[JAMIE]
Open source is human, so difficult situations happen. What guidance do learners get?

[ALEX]
Several scenarios. If you receive harsh feedback, slow down, extract the actionable content, and respond to the substance rather than mirroring the tone. If you disagree with a decision, ask clarifying questions and state your reasoning calmly, but respect maintainers’ project authority. If someone is rude, rely on documented community standards and, if needed, ask a moderator or maintainer for help. If you accidentally caused offense, acknowledge it, apologize directly, and adjust.

[JAMIE]
That’s a reassuring framework. Not every awkward moment means you failed.

[ALEX]
Exactly. Repair is part of collaboration.

[PAUSE]

[JAMIE]
Now I want to connect this to documentation. Why is README writing in a culture chapter?

[ALEX]
Because documentation is hospitality. A good README tells people what the project is, how to install it, how to use it, how to contribute, and what license governs it. A simple skeleton includes project name, installation, usage, contributing, and license.

[JAMIE]
And accessibility belongs there too.

[ALEX]
Absolutely. Accessible READMEs use meaningful headings, descriptive link text, clear structure, and concise language. A good README helps a newcomer get started. A bad README assumes context they do not have.

[JAMIE]
Related to that are community health files.

[ALEX]
Yes: `CONTRIBUTING.md`, `CODEOFCONDUCT.md`, `SECURITY.md`, and `LICENSE`. These files help contributors know how to participate, how the community behaves, how to report security issues, and what reuse is permitted. You should learn to find these on GitHub because they shape how you engage with a repository.

[JAMIE]
And different communication channels matter too.

[ALEX]
Right. Use issues for trackable work, PR comments for change-specific discussion, discussions or forums for broader conversation if the project supports them, and security channels for sensitive vulnerabilities. Choosing the right channel is part of professional communication.

[PAUSE]

[JAMIE]
There’s also broad newcomer guidance in the chapter: what open source is, who can contribute, and how to find a first contribution.

[ALEX]
Yes, and it’s worth saying plainly: open source contribution is not limited to writing code. People contribute documentation, design, testing, issue triage, accessibility review, translation, examples, and thoughtful feedback. A good first contribution is usually small, well-scoped, and understandable. Warning signs that it’s too large include many files, unclear requirements, missing reproduction steps, or lots of unresolved discussion.

[JAMIE]
So before starting, read the issue carefully.

[ALEX]
Exactly. Understand the problem, the expected fix, any linked discussion, and whether someone is already working on it. Then follow the basic workflow: fork if needed, clone, create a branch, edit files, commit, push, and open a PR. A good PR description explains what changed, why, and how to test it.

[JAMIE]
And after a contribution is merged?

[ALEX]
Thank people, sync your fork, clean up your branch, and reflect on what you learned. Over time, build a contribution habit: small regular participation, thoughtful reading, respectful review, and realistic scope.

What does the PR tree usually contain?

And you can inspect PR details there too.

Now checking out a PR branch. Why do that instead of just reading on the website?

And then return to your original branch when you’re done.

[ALEX]
Exactly. That’s an important cleanup step. Don’t stay accidentally parked on someone else’s PR branch.

Let’s get to diff review in VS Code, because that’s where accessibility really matters.

[ALEX]
In the PR detail view, you can open the list of changed files. Opening one shows a diff. By default, many people see split view, old content on the left and new on the right. Inline view presents changes in one continuous flow. You can toggle between them.

But for screen reader users, raw visual diff layouts can be exhausting.

[ALEX]
That’s exactly why the Accessible Diff Viewer matters. In VS Code, `F7` moves to the next change and `Shift+F7` moves to the previous change. [JAMIE]
So it turns “maze of symbols” into “readable change units.”

[ALEX]
Beautifully said. The chapter strongly recommends that workflow for screen reader users because it is far superior to line-by-line navigation of the visual diff.

And if I find an issue?

[ALEX]
You can flag it during review with an inline comment. Put the cursor on the relevant line and use the command to add a comment. If that is difficult in VS Code, fall back to GitHub.com. Accessibility includes using the environment that lets you work accurately.

[JAMIE]
What about creating a pull request from VS Code instead of in the browser?

[ALEX]
The extension supports that too. Prerequisites are that your branch exists with committed changes and is pushed or ready to push. You can create a PR through the Command Palette, the Source Control panel, or the GitHub panel. The Command Palette method is often the clearest.

Then you fill out a form.

[ALEX]
Right. Title is required. Description is optional but strongly recommended. You choose the base branch, usually `main`, and the compare branch, your feature branch. [JAMIE]
And there are different submission styles.

[ALEX]
Yes. Some workflows prompt interactively for title, body, and base branch. Others let you provide details inline. You may also create as draft or open the form in your browser if that’s easier. Again, accessibility means choosing the route that works best for you.

[PAUSE]

[JAMIE]
PR description templates can make this a lot easier, especially for beginners.

[ALEX]
Definitely. Repositories may store PR templates so when you open a PR, a structure appears automatically. Common sections include description, related issue, type of change, testing, checklist, and screenshots if applicable.

[JAMIE]
That’s helpful because it turns “What do I write?” into prompts.

[ALEX]
Exactly. For screen reader users, working through the template in order can also reduce cognitive load. Read each prompt, replace placeholders with real details, and leave out irrelevant sections only if the project expects that.

[PAUSE]

[JAMIE]
Now the formal review actions: commenting, approving, and requesting changes.

[ALEX]
When adding an inline comment, be specific and tie your note to the exact changed line. If you have multiple comments, use Start Review so they can be submitted together instead of sending a stream of isolated notifications. When ready, submit the review as one of three verdicts: approve, request changes, or comment-only.

[JAMIE]
And each verdict carries different weight.

[ALEX]
Right. Approve means the PR is ready from your perspective. Request changes means there are issues that should be addressed before merge. Comment-only means feedback without a formal accept-or-block judgment.

[JAMIE]
There was also mention of viewing PR diffs in your terminal.

[ALEX]
Yes, in some environments you may use CLI tooling to inspect PR diffs or metadata from the terminal. That can be another accessibility or efficiency path, depending on the learner.

[JAMIE]
Merging is the next step. What should learners know?

[ALEX]
Before merging, confirm prerequisites like reviews, passing checks, and project permissions. You may merge from the PR detail view or command palette in VS Code, depending on access. Common merge types include the default merge strategy and squash-and-merge. [JAMIE]
And sometimes auto-merge is available.

[ALEX]
Yes. Auto-merge can be enabled so the PR merges automatically when required checks pass. After merging, switch back to main and pull the latest changes. Then delete the feature branch if the project’s workflow calls for it.

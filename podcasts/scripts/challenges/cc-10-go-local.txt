Before we walk the tools, let’s build the mental model. Git makes more sense when you know its three areas: the working directory, the staging area, and the repository.

[JAMIE]
I hear those terms a lot, especially staging area, and I think that’s where many people start nodding politely instead of actually understanding.

[ALEX]
Fair. So let’s make it concrete. The working directory is the actual project folder on your computer. In VS Code, it’s the files you open in Explorer. If you edit a file there, you’ve changed the working directory.

[JAMIE]
But changing a file does not automatically become a commit.

[ALEX]
Exactly. Git notices the change, but it does not save it to history yet. The staging area, also called the index, is the holding zone for the next commit. When you stage a file, you’re telling Git, yes, include this version in my next snapshot.

[JAMIE]
And then the repository is where committed history lives.

[ALEX]
Yes. The repository is Git’s database, stored in the hidden .git folder. Once you commit, Git records a snapshot of what was staged. So the flow is: edit in the working directory, stage what belongs together, then commit that staged snapshot into the repository.

[JAMIE]
So edit, stage, commit. Working directory, staging area, repository.

[ALEX]
Perfect. One useful analogy is packing a box. Your desk is the working directory. The open shipping box is the staging area. Sealing and labeling the box is the commit. You can add or remove items from the box before sealing it.

[JAMIE]
That actually helps. Especially because it explains why unstaging is not the same as deleting.

[ALEX]
Exactly right.

[JAMIE]

[JAMIE]
Okay, now what exactly is a commit?

[ALEX]
A commit is a snapshot of your project at a moment in time. It includes the staged content, a human-readable message, the author identity, and a pointer to the previous commit.

[JAMIE]
So it’s more than “here are the lines that changed.”

[ALEX]
Right. Git can show differences between commits, but a commit itself records a full project state for the tracked files involved in that snapshot. Each commit also gets a unique ID called a hash, often shortened to a seven-character form when you’re reading logs.

[JAMIE]
And commits are mostly permanent.

[ALEX]
That’s the big safety feature. You can add more commits to fix mistakes, revert changes, or move around in history, but the original commit becomes part of the timeline. That permanence is what makes Git such a reliable record.

[JAMIE]
For screen reader users, is there a good way to confirm a commit happened?

[ALEX]

[PAUSE]

[JAMIE]
And branches?

[ALEX]
A branch is simply a movable label pointing to a series of commits. The default branch is usually `main`. When you create a new branch, you create a safe place to work without changing `main` directly.

[JAMIE]
So a branch is not a full copy of the repository in the everyday sense.

[ALEX]
Correct. It’s lightweight. You create the branch, then as you make commits on it, that branch advances. `HEAD` tells Git which branch, or more precisely which current checkout, you are on right now.

[JAMIE]

[ALEX]
Exactly. And that matters a lot. One of the best habits you can build is checking your current branch before you edit or commit.

[JAMIE]

[ALEX]
Yes. Local and remote are related, but they are not automatically synced every second. You make changes and commits locally. Then you push to send your commits to GitHub. [JAMIE]

Before the step-by-step, can we talk about merge conflicts? People get nervous the moment that phrase appears.

[ALEX]

[JAMIE]

[ALEX]
Exactly. Conflicts are a routine collaboration event. [ALEX]

[JAMIE]

[ALEX]

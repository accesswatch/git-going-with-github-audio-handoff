## What this changes
One sentence describing the change.

## Why
One sentence explaining why it matters.

Closes #XX

[JAMIE]

[ALEX]

## What this changes

## Why
Students need a working link to find session times. The TODO was a known gap filed in issue number 3.

Closes #3

That’s nice because a reviewer can immediately understand the file, the exact type of fix, and the reason.

[JAMIE]
So it includes metadata, discussion, and code comparison.

[ALEX]
Yes. And that’s why learning to navigate it accessibly matters.

[PAUSE]

[JAMIE]

[ALEX]

[JAMIE]

[ALEX]

The key idea is that you can comment on a specific span of changed lines, not just leave a general note.

[JAMIE]

[ALEX]

Can we talk about draft pull requests?

[ALEX]
Yes. A draft PR is useful when you want to share work early but are not yet ready for formal review or merge. It signals “please don’t treat this as finished.”

[JAMIE]
When should someone use a draft?

[ALEX]

[JAMIE]
What does a draft PR do differently?

[ALEX]
It changes the PR state so reviewers know it is not ready for approval in the usual way. It may limit merge actions until it is marked ready for review. It’s a communication tool as much as a technical status.

[JAMIE]
And later, the author can mark it ready for review.

[ALEX]
Exactly. There is a control to convert draft to ready. There’s also typically a path to convert an open PR back into draft if needed. In CLI-oriented workflows, you can also create a draft PR, check whether a PR is in draft state, and list only draft PRs.

So draft is not a dead end. It’s part of the lifecycle.

[ALEX]
Right.

What about requesting reviewers?

[ALEX]

Now let’s switch perspectives again. Suppose someone is a maintainer. What merge options should they understand?

[ALEX]
Three common options are merge commit, squash and merge, and rebase and merge. A merge commit preserves the branch history as a distinct merge event. Squash and merge combines the PR’s commits into one commit on the target branch. Rebase and merge reapplies commits onto the target branch without a merge commit.

[JAMIE]
And there’s also the choice to delete the branch afterward.

[ALEX]
Yes. Deleting the branch after merging helps keep the repository tidy, especially for short-lived feature branches.

[JAMIE]
And auto-merge?

[ALEX]
Auto-merge lets a PR merge automatically once required conditions are met, such as approvals and passing checks. It’s useful when everything is ready except a pending check and you do not want to wait around watching the page.

[JAMIE]
How does a user enable it?

[ALEX]
On the PR page, if the repository allows it and the necessary prerequisites are in place, there is an option to enable auto-merge. Screen reader users should navigate the action controls near merge options. Once enabled, GitHub waits for requirements to pass, then merges automatically.

[JAMIE]
And if circumstances change?

[ALEX]
You can cancel auto-merge before it completes.

What happens after a PR is merged?

[ALEX]

[JAMIE]

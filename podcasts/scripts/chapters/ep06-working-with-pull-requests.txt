[JAMIE]
Let’s answer the basic question directly: what is a pull request?

[ALEX]
A pull request is a proposal to merge a set of changes from one branch into another, usually into main. It bundles the code or content changes together with a description, conversation, review activity, status checks, and merge controls.

[JAMIE]
So it’s part diff, part discussion, part gatekeeping mechanism.

[ALEX]
Well put. It gives reviewers context, gives automation a place to run, and gives maintainers a decision point.

Let’s talk draft pull requests, because people often hear about them but don’t know when to use them.

[ALEX]
A draft PR is for work you want visible but not yet fully reviewable. Use a draft when you want early feedback, when checks are still being set up, when the change is incomplete, or when you want collaborators to know the work is in progress without implying it’s ready to merge.

[JAMIE]
So what does draft status change?

[ALEX]
It changes expectations. A draft PR is marked clearly as draft, and reviewers understand that formal review may wait. It cannot be treated the same as a ready PR for merge flow. When it becomes mature enough, you mark it ready for review.

[JAMIE]

[JAMIE]
Now let’s talk maintainers and merge options.

[ALEX]
There are usually three common merge strategies. A regular merge commit preserves the branch history and adds a merge commit. Squash and merge combines the branch’s commits into one commit on the target branch. Rebase and merge replays the commits onto the target branch without a merge commit.

[JAMIE]
How should a learner think about those?

[ALEX]
Merge commit preserves more of the branch structure. Squash is often preferred for small or messy branches because it creates a clean final history. Rebase and merge keeps a linear history but still preserves individual commits. Projects choose among these based on their workflow standards.

[JAMIE]
And there’s usually an option to delete the branch afterward.

[ALEX]
Yes. Deleting the branch after merging is often a good cleanup step for short-lived feature branches. It reduces clutter and signals that the work is complete.

[JAMIE]
And auto-merge?

[ALEX]
Auto-merge lets a PR merge automatically once required conditions are satisfied, such as passing checks and required reviews. It’s useful when you don’t want to sit watching for the exact moment a PR becomes mergeable.

[PAUSE]

[JAMIE]
So after a PR is merged, what should happen next?

[ALEX]
Confirm that the linked issue closed if you used the closing pattern correctly. Pull or sync the updated target branch locally if needed. Clean up the feature branch if it still exists. And mentally close the loop: the contribution has moved from proposal to accepted history.

[JAMIE]
That closure matters, especially for beginners.

[ALEX]
It does.

[PAUSE]

[JAMIE]

With a screen reader, locate the auto-merge option in the merge area of the PR page. After enabling it, GitHub waits for required checks and approvals, then merges when conditions are met.

[JAMIE]

[ALEX]

The practical scenarios section is helpful because it turns all this into actual decision-making. Scenario one: I want to review an assigned PR.

[ALEX]
Start by locating the PR from your notifications, assignment context, or the PR list. Read the description first to understand intent. Check status checks so you know whether automation already found problems. Open Files changed and review the diff. Add inline comments if needed. [JAMIE]
Scenario two: I want to respond to review feedback on my own PR.

[ALEX]

[JAMIE]

[ALEX]
That means your branch and the target branch now disagree in a way GitHub cannot merge automatically. You’ll need to update your branch, often by merging or rebasing the latest main into it locally, resolving the conflict, then pushing the result. The key lesson is not to panic: a conflict is a synchronization problem, not a moral failing.

[ALEX]
Welcome back. This time we’re focusing on the part of open source that isn’t just about commands or files. It’s about people. Technical skill can help you make a contribution, but communication skill is what helps you work with a community, get reviews, ask for help, and keep contributing.

[JAMIE]
I’m glad we’re doing this one. A lot of learners can follow steps, but then freeze when it’s time to comment on an issue, open a pull request, or respond to feedback in public.

[ALEX]
Exactly. And this chapter also includes workshop logistics, so let’s make those clear. There’s one Chapter 8 challenge, and it’s a guided reflection, not a bot-graded task. There’s no automation check because communication quality is too subjective for fair automatic scoring.

[JAMIE]
So what do learners actually submit?

[ALEX]
A reflection comment on their assigned Chapter 8 challenge issue in their Learning Room repository on GitHub. That comment is the evidence. No extra file, no screenshot, no separate upload. When the comment is posted, the facilitator reviews it for specificity, and then the learner closes the issue.

[JAMIE]
And the reflection isn’t just, “I’ll try to be respectful,” right?

[ALEX]
Right. It needs three specific, actionable behaviors. The format is:
Chapter 8 reflection.
One respectful review habit I will use.
One way I will ask for help clearly.
One way I will respond to feedback constructively.
Each answer should be a real behavior you can do today.

[JAMIE]
Like, “I will start review comments by naming something that works before suggesting a change,” instead of “I will be nice.”

[ALEX]
Perfect example. Or, “I will explain the exact step where I got stuck and what I already tried,” instead of “I will ask better questions.”

[PAUSE]

[ALEX]
The step-by-step process is simple. Read the chapter, think about a real communication moment from earlier work, open your assigned issue, go to the comment box, write the reflection using the required prompts, and activate the Comment button, or use Control+Enter if that works better for you.

[JAMIE]
And if someone gets stuck writing the reflection?

[ALEX]
Keep it simple. One sentence per prompt is enough. Draft in a text editor first if that’s more accessible or less stressful. Revisit the sections on feedback and asking for help for examples. And if needed, ask a facilitator for a sample response, then adapt it in your own words.

[JAMIE]
So the learning goal here is really behavior change.

[ALEX]
Yes. The expected outcomes are that learners can name respectful collaboration behaviors, prepare for constructive review work later, and feel safer asking for help in public threads. The deeper lesson is that technical quality and communication quality reinforce each other. Clear, respectful communication helps good work move faster.

[JAMIE]
And the pattern of learning here is also deliberate.

[ALEX]
It is. First, absorb community norms and the reasons behind them. Second, reflect on your own experience. Third, commit to behaviors in writing. Fourth, apply those behaviors later in reviews, comments, and pull requests.

[PAUSE]

[ALEX]
Now, before we talk about etiquette, we need the workflow that those conversations sit inside: GitHub Flow.

[JAMIE]
Because comments and reviews make more sense when you understand what a branch and a pull request are doing socially, not just technically.

[ALEX]
Exactly. GitHub Flow is the standard lightweight contribution workflow. It works well in open source because it stays simple. The six steps are: create a branch from main, make changes and commit them, open a pull request, discuss and review, pass status checks, and merge.

[JAMIE]
Let me try to say that back in everyday language. Start a branch for one task, do the work in small logical commits, open a PR even before everything is perfect if you need early feedback, respond to comments, make sure tests and checks pass, and then a maintainer merges it.

[ALEX]
That’s it. And there are a few ideas behind why this works. First, main stays deployable. You don’t commit directly to main in a healthy collaborative project. Changes go through pull requests and review. Second, branches are cheap. You make one for a task and delete it after merge. Third, the pull request becomes the unit of conversation. The why, the tradeoffs, the test notes, the feedback, the approval, all in one place. And fourth, small changes move faster.

[JAMIE]
That last one matters a lot. A tiny pull request is easier to review, easier to understand, and usually less emotionally intimidating.

[ALEX]
Exactly. A focused PR also supports accessibility. It reduces cognitive load for reviewers, including blind and low-vision contributors using screen readers who may need to navigate comment by comment and file by file.

[JAMIE]
Now I know some learners will hear “Git Flow” somewhere else and wonder if it’s the same thing.

[ALEX]
Good distinction. Git Flow is a different branching model, older and more complex. It uses multiple branch types: a permanent main branch for production releases, a permanent develop branch for upcoming work, temporary feature branches, release branches, and hotfix branches.

[JAMIE]
So instead of branching from main and coming back to main, feature work usually branches from develop and merges back into develop.

[ALEX]
Right. Then when enough work is ready, a release branch is created for final testing and version updates. That release merges into main and back into develop. If a production bug appears, a hotfix branch starts from main and then merges to both main and develop.

[JAMIE]
That sounds useful in some environments, but heavy for a workshop.

[ALEX]
That’s the key point. GitHub Flow differs by having only one long-lived branch, main. Feature work branches off main and returns to main through a PR. Releases are continuous instead of centered on release branches. Hotfixes are handled as normal PR work. So GitHub Flow is lower complexity and is especially good for open source, web projects, and continuous deployment.

[JAMIE]
Where might learners still encounter Git Flow?

[ALEX]
In enterprise software with scheduled releases, mobile apps with app store review, embedded or firmware work, or older legacy teams that adopted it before continuous deployment became common.

[JAMIE]
And this workshop uses GitHub Flow because it maps well to how open source contribution usually works.

[ALEX]
Yes. It’s easier to learn, easier to practice, and easier to collaborate with across many contributors.

[PAUSE]

[JAMIE]
There’s also an unwritten rule that people often learn the hard way: one thing per branch.

[ALEX]
Yes. A branch should represent one coherent change. If you are fixing alt text, don’t also refactor unrelated code and update dependency versions in the same branch. One thing per branch leads to clearer commits, smaller pull requests, easier reviews, and fewer merge conflicts.

[JAMIE]
So even if I notice three unrelated problems, I shouldn’t treat one branch as my personal cleanup spree.

[ALEX]
Exactly. Separate branches create separate conversations.

[PAUSE]

[ALEX]
Another practical skill in open source is keeping your fork up to date.

[JAMIE]
Because if your fork drifts behind the original repository, your future work starts from stale code.

[ALEX]
Right. You sync your fork so your copy of main matches the upstream project. That reduces conflicts, helps you branch from current code, and prevents confusing pull requests based on outdated history.

[JAMIE]
What’s the easiest way?

[ALEX]
On GitHub’s web interface, many forks show a Sync fork control. That’s often the easiest method. Activate it, review what will sync, and update your fork.

[JAMIE]
And if someone is using Git in the VS Code terminal?

[ALEX]
Then there’s a one-time setup step: add the upstream remote, meaning the original repository, not your fork. Typically that means a command like git remote add upstream followed by the original repository URL.

[JAMIE]
So origin is your fork, upstream is the project you forked from.

[ALEX]
Exactly. After that, the sync process is four steps. First, switch to your main branch. Second, fetch upstream changes. Third, merge upstream’s main into your own main. Fourth, push your updated main back to your fork on GitHub.

[JAMIE]
And only after your main is current do you create a new feature branch.

[ALEX]
Yes. That’s a healthy rhythm. Sync before starting new work, and sync again if a lot has changed while you were away. You may also sync before opening a PR if the project moved quickly and you want to reduce surprises.

[JAMIE]
There’s also GitHub Desktop for people who prefer a graphical app.

[ALEX]
Right. That can be a comfortable option for some learners. The principle is the same: update your fork from upstream so your main branch stays current.

[PAUSE]

[ALEX]
Let’s move to commit messages, because they are a form of communication too.

[JAMIE]
This is where people often type “stuff” or “updates” and regret it later.

[ALEX]
Exactly. A good commit message tells future readers what changed and why. The common structure has three parts: a first line, which is required; an optional body; and an optional footer.

[JAMIE]
The first line is the headline.

[ALEX]
Yes. It should be short, clear, and specific. Something like, “Add alt text to workshop screenshots,” or, “Fix broken README link.” The body is optional, but useful when context matters. It can explain why the change was needed, what tradeoff you made, or any detail a reviewer would want later. The footer is also optional and often used for issue references or structured metadata.

[JAMIE]
So for many small changes, a strong first line might be enough.

[ALEX]
Correct. But whatever you write, avoid vague messages like “fixes,” “more changes,” or “final version.” Those don’t help reviewers, maintainers, or your future self.

[JAMIE]
And this ties into atomic commits.

[ALEX]
It does. An atomic commit is one coherent, complete change. Not half a fix. Not three unrelated fixes. One logical unit. Atomic commits are easier to review, revert, understand, and explain.

[JAMIE]
So common mistakes would include mixing unrelated work in one commit, making giant catch-all commits, or writing messages that don’t describe the actual change.

[ALEX]
Exactly. Good commit messages in practice are specific, action-oriented, and aligned with the actual contents of the commit.

[PAUSE]

[ALEX]
Now let’s widen the lens. Open source communication has three characteristics that shape everything else.

[JAMIE]
In writing, asynchronously, and in public.

[ALEX]
Exactly. In writing means your words persist. People may reread them later, quote them, or interpret tone without hearing your voice. Asynchronously means the other person might not reply for hours or days, and silence doesn’t automatically mean rejection. In public means many conversations happen where others can see them, learn from them, and join them.

[JAMIE]
That changes how we should phrase things. You can’t rely on facial expression or instant clarification.

[ALEX]
Right. Clarity matters more. Specificity matters more. And kindness matters more because text strips away a lot of context.

[PAUSE]

[JAMIE]
Let’s talk about feedback. What makes feedback actually helpful?

[ALEX]
There’s a strong five-part pattern. First, acknowledge what’s working. That shows respect and gives the author a stable point to build from. Second, identify the specific concern. Not “this is confusing,” but “the installation step assumes Node is already installed.” Third, explain why it matters. Maybe it affects accessibility, maintainability, accuracy, or user success. Fourth, suggest a path forward when you can. And fifth, signal the weight of the concern.

[JAMIE]
That last part is really useful. Not every comment has the same force.

[ALEX]
Exactly. A reviewer should indicate whether something is required, recommended, or just a personal preference. That prevents authors from treating every sentence as a blocker.

[JAMIE]
Can we model the difference?

[ALEX]
Sure. Less helpful: “Change this.” More helpful: “Nice clarification in this section. I think the image also needs alt text so screen reader users receive the same information. Could we add a brief description here? I’d treat this as important before merge.”

[JAMIE]
That gives praise, names the issue, explains why, offers a fix, and signals the concern level.

[ALEX]
Exactly.

[PAUSE]

[ALEX]
Language and tone also matter. A strong default is to prefer “we,” or describe the code or document, not the person.

[JAMIE]
So instead of “You did this wrong,” say, “This function may fail when the input is empty,” or, “We may want to handle the empty state here.”

[ALEX]
Yes. That lowers defensiveness and keeps the focus on improving the work. Another good habit is using tentative language when you’re uncertain. Phrases like, “I think,” “I may be missing context,” “Could we,” or, “Would it make sense to…” can signal openness.

[JAMIE]
Especially when you’re not a maintainer or you’re still learning.

[ALEX]
Right. Also remember cultural and language diversity. Not every contributor reads or writes English the same way. So avoid sarcasm, idioms, and overly compressed shorthand if clarity matters. Write plainly.

[JAMIE]
And avoid fake urgency.

[ALEX]
Exactly. Don’t mark something urgent unless it truly is. Urgency markers can create stress, especially in volunteer communities. If something is genuinely urgent, say why.

[PAUSE]

[JAMIE]
Now for commenting etiquette, because even people with good intentions can make review threads messy.

[ALEX]
Absolutely. First, keep comments focused. One comment should usually address one concern. That makes it easier to reply, resolve, and track. Second, don’t leave comments unresolved if the issue is actually settled. If the author changed the code and the concern is handled, resolve the conversation.

[JAMIE]
And resolving conversations is not just tidying up. It’s communication.

[ALEX]
Yes. It tells everyone, including future readers, that this point has been addressed. Third, do not pile on. If another reviewer already raised the same issue, add value only if you have new information. Repeating the same criticism can feel overwhelming.

[JAMIE]
What about reactions?

[ALEX]
They’re useful. A thumbs up or similar reaction can signal agreement without creating noise. That’s often better than posting “same” or “agree.”

[JAMIE]
There’s also a really practical accessibility tip here: saved replies.

[ALEX]
Yes. Saved replies are a major accessibility and efficiency win. If you often write similar comments, like requesting tests, asking for reproduction steps, thanking contributors, or pointing to contribution guidelines, you can store a reusable response. That reduces repetitive typing, lowers fatigue, and helps keep tone consistent.

[JAMIE]
What are common uses?

[ALEX]
Welcoming first-time contributors, asking for more detail in a bug report, explaining review timelines, requesting screenshots or test notes, or giving a standard accessibility reminder in respectful language.

[JAMIE]
And how do you create one?

[ALEX]
In GitHub settings, there’s a Saved replies area where you can create and name a reusable reply. Then, when writing a comment, you can insert that saved reply into the comment box.

[JAMIE]
And for screen reader users, the important part is that there is a navigable path to the saved replies controls and insertion options, so this can support efficient commenting without copy-paste hunting.

[ALEX]
Exactly. The details of the interface may shift over time, but the concept remains: use saved replies to reduce effort and improve consistency.

[PAUSE]

[ALEX]
Let’s split code review etiquette into two sides: reviewers and authors.

[JAMIE]
Start with reviewers.

[ALEX]
Review the code, not the person. Don’t gatekeep knowledge by acting like unfamiliarity is a flaw. Ask questions instead of making demands when there’s room for discussion. Distinguish opinion from requirement. And when the work is ready, approve explicitly.

[JAMIE]
That “don’t gatekeep knowledge” point is important. A review should not say, “You should already know this.”

[ALEX]
Exactly. Better to explain briefly, link to a resource, or name the convention being applied. Reviews should help the project and help people grow.

[JAMIE]
And distinguishing opinion from requirement could sound like, “Required: this needs alt text for accessibility,” versus, “Optional: I personally prefer this variable name.”

[ALEX]
Exactly right.

[JAMIE]
What about for authors?

[ALEX]
Say thank you. Don’t take feedback personally. Explain your choices when context matters. And surface blockers early.

[JAMIE]
So if I can’t reproduce a suggested fix, or I’m confused by a review comment, I shouldn’t silently stall.

[ALEX]
Correct. Respond clearly. Say what you tried, what you’re uncertain about, and where you’re blocked. That makes collaboration easier and keeps momentum.

[PAUSE]

[JAMIE]
This chapter also calls out inclusive commenting for accessibility issues.

[ALEX]
Yes, and that matters a lot in this course. When raising accessibility concerns, focus on the user impact and the fix, not on blame. For example, instead of, “You forgot accessibility again,” try, “This image appears to be missing alt text, which means screen reader users may miss the information here. Could we add a short description?”

[JAMIE]
So accessibility feedback should be firm when needed, but still respectful and explanatory.

[ALEX]
Exactly. Accessibility is not a cosmetic preference. It often belongs in the required category. But the tone can still be inclusive and collaborative.

[PAUSE]

[ALEX]
There’s also something called the social contract around “good first issue.”

[JAMIE]
Meaning that label creates expectations.

[ALEX]
Yes. If a maintainer labels something “good first issue,” the issue should be reasonably scoped, documented clearly enough for a newcomer to attempt, and not hide major unstated complexity. And if you’re a contributor picking one up, you should still read carefully, ask clarifying questions when needed, and respect that someone else may already be working on it.

[JAMIE]
So the label is an invitation, not a guarantee of zero difficulty.

[ALEX]
Exactly. But it should be a sincere invitation.

[PAUSE]

[JAMIE]
What about difficult situations? Because eventually everyone runs into one.

[ALEX]
Yes. If you receive harsh feedback, pause before replying. Separate the technical point from the tone. Respond to the substance calmly, and if needed, ask for clarification. If you disagree with a decision, state your reasoning respectfully, ask if there are project constraints you may not see, and accept that maintainers make final calls in their projects.

[JAMIE]
And when someone is rude to you?

[ALEX]
Don’t escalate in the heat of the moment. Point back to the issue at hand, stay factual, and use the project’s code of conduct or moderation channels if needed. You do not have to absorb mistreatment as a price of participation.

[JAMIE]
And if you accidentally caused offense?

[ALEX]
Acknowledge it plainly. Don’t get defensive. Apologize, restate your intent if helpful, and then adjust your behavior. In open source, repair matters more than perfection.

[PAUSE]

[ALEX]
Let’s switch to documentation, starting with your first README.

[JAMIE]
Because a README is often the first conversation a project has with a newcomer.

[ALEX]
Exactly. A good README helps someone understand what the project is, how to install it, how to use it, how to contribute, and what license applies.

[JAMIE]
So the basic skeleton is: Project Name, Installation, Usage, Contributing, and License.

[ALEX]
Yes. Project Name identifies the project clearly. Installation explains how to get it running. Usage shows how to actually use it. Contributing tells people how to participate. License explains the legal terms.

[JAMIE]
What belongs in each section in practice?

[ALEX]
For Project Name, usually a short explanation of what the project does. For Installation, prerequisites and commands or steps. For Usage, examples, expected behavior, maybe screenshots if they’re accessible. For Contributing, branch workflow, issue or PR expectations, testing steps, and links to CONTRIBUTING.md if there’s a fuller guide. For License, the license name and ideally a link to the license text.

[JAMIE]
And accessibility in READMEs?

[ALEX]
Very important. Use meaningful headings, descriptive link text, accessible image alt text, and plain language. Avoid giant unstructured walls of text. If you include screenshots, don’t let the screenshot carry essential information without a text equivalent.

[JAMIE]
So a bad README might be vague, missing setup steps, full of unexplained jargon, and inaccessible in structure.

[ALEX]
Exactly. A good README helps a stranger succeed.

[PAUSE]

[JAMIE]
That connects to community health files too.

[ALEX]
It does. Community health files tell contributors how the project works. CONTRIBUTING.md explains how to contribute. CODEOFCONDUCT.md defines behavior expectations and reporting paths. SECURITY.md explains how to report vulnerabilities safely. LICENSE states the legal permissions and restrictions.

[JAMIE]
And finding these files on GitHub is a practical skill.

[ALEX]
Yes. Learners should know to look in the repository root, in a .github folder, or in GitHub’s community standards surfaces where these files may be highlighted. Reading them before contributing saves time and helps avoid social mistakes.

[PAUSE]

[ALEX]
Another important topic is choosing the right communication channel.

[JAMIE]
Because not every question belongs in the same place.

[ALEX]
Right. Use issues for bugs, feature requests, and trackable discussions. Use pull request comments for feedback tied to a specific change. Use discussions, if enabled, for broader questions or ideas. Use security channels for vulnerabilities, not public issues. And use synchronous channels like chat sparingly and with awareness that important decisions should usually be summarized back in a public, durable place.

[JAMIE]
That “durable place” idea matters in open source. If the answer only happens in a private message, future contributors can’t benefit.

[ALEX]
Exactly.

[PAUSE]

[JAMIE]
The chapter also gives quick phrases that work well.

[ALEX]
Yes, and they’re worth practicing. Phrases like, “I may be missing context, but…”, “Could we…”, “One thing that’s working well here is…”, “I’m blocked at this step…”, “Here’s what I tried…”, “Would you prefer A or B?”, or, “I think this may need alt text because…” These phrases keep communication specific, calm, and collaborative.

[JAMIE]
And there’s a practical exercise: rewrite one comment.

[ALEX]
Right. Take a blunt or vague comment and turn it into one that is specific, respectful, and actionable. That’s excellent practice because tone is a skill, not just a personality trait.

[PAUSE]

[ALEX]
Now let’s zoom out to contributing to open source more generally, especially for first-time contributors.

[JAMIE]
Starting with the most basic question: what is open source?

[ALEX]
Open source means the source code is publicly available under a license that allows people to inspect, use, modify, and share it under defined terms. It’s not just code posted online. The license matters.

[JAMIE]
And who can contribute?

[ALEX]
Anyone who follows the project’s rules and norms. Contributions aren’t limited to code. Documentation, issue triage, testing, accessibility improvements, design, examples, bug reports, and review are all valuable.

[JAMIE]
What makes a good first contribution?

[ALEX]
Something small, clear, and realistically finishable. Good first contributions often improve docs, fix typos, add missing alt text, improve labels, clarify setup steps, or resolve a well-scoped bug.

[JAMIE]
And signs a first contribution is too large?

[ALEX]
If it spans many files, changes architecture, requires deep project knowledge, has unclear requirements, or depends on unresolved design decisions, it may be too large for a first attempt.

[JAMIE]
How do you find something to work on?

[ALEX]
Look for labels like good first issue, help wanted, documentation, accessibility, or bug. Read recent issues and pull requests. Check whether the project is active and whether maintainers respond. Start where the path is visible.

[JAMIE]
Then before starting, read the issue carefully.

[ALEX]
Yes. Make sure the problem is still relevant, see whether anyone has claimed it, read linked discussion, and understand any constraints. If needed, leave a brief comment saying you’d like to work on it.

[PAUSE]

[JAMIE]
Then comes making the contribution itself.

[ALEX]
Right. The tool sequence is familiar: fork the repository if you don’t have write access, clone your fork locally if you’re working on your machine, create a branch, edit files, commit your changes, push the branch, and open a pull request.

[JAMIE]
And the basic workflow should align with GitHub Flow.

[ALEX]
Exactly. One task, one branch, clear commits, then a PR.

[JAMIE]
What makes a good PR description?

[ALEX]
Explain what changed, why it changed, how to test it, and any context reviewers need. Link the related issue when appropriate. If there are limitations or open questions, say so. A good PR description reduces back-and-forth and helps reviewers review efficiently.

[JAMIE]
Then there’s getting help.

[ALEX]
Yes. Ask in the most appropriate public place when possible. Be specific. Include the exact step where you got stuck, error messages if any, what you expected, and what you already tried. Good help requests make it easier for others to help you.

[JAMIE]
And after the contribution is merged?

[ALEX]
Say thank you, delete the merged branch if appropriate, sync your fork again, and stay open to follow-up learning. Your first merged contribution is not the end. It’s the start of familiarity.

[JAMIE]
Which leads to building a contribution habit.

[ALEX]
Exactly. Practical habits include keeping contributions small, reading project docs first, syncing your fork regularly, asking clear questions, responding graciously to review, and coming back consistently rather than waiting for a “perfect” giant contribution.

And if writing in GitHub’s comment box feels awkward, draft somewhere else first, then paste it in.

[ALEX]

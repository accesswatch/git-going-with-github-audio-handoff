[ALEX]
Let’s start with cherry-pick. Cherry-pick means: take one specific commit from somewhere else and apply that commit onto the branch you’re on now.

[JAMIE]
So not merge the branch. Not rebase the branch. Just copy one change.

[ALEX]
That’s the idea. Think of it as surgically lifting one commit out of another branch. You’d use it when a bug fix exists on a feature branch and main needs that fix immediately, or when a teammate has one useful commit but you don’t want all their in-progress work, or when you committed to the wrong branch and need that commit somewhere else.

[JAMIE]
Before you can cherry-pick, you need the commit identifier, the SHA.

[ALEX]
Right. In VS Code, one path is the Timeline view. You can open the Timeline, move to the branch that contains the commit, inspect the history there, and copy the SHA from the details.

[JAMIE]
And from the terminal?

[ALEX]
A very direct command is to view another branch’s history without switching to it: git log, the branch name, and oneline output. That gives you short commit IDs followed by commit messages. So you might see something like a short SHA, then “Fix null pointer error in auth module,” then another SHA for “Add unit tests for login.” The short SHA is typically the first seven characters, and that’s often enough for local operations.

[JAMIE]
That “without switching to it” part is nice. People often think they have to check out the other branch just to inspect it.

[ALEX]
You usually don’t.

[JAMIE]
Okay. Once I have the SHA, how do I actually do the cherry-pick?

[ALEX]
In VS Code, open the Command Palette, type cherry, choose the Git cherry-pick command, paste or type the SHA, and VS Code applies it to your current branch.

[JAMIE]
And gh?

[ALEX]
GitHub CLI does not itself perform cherry-pick, because cherry-pick is a local Git operation. But gh can help you discover the commit SHA through the GitHub API. Then you still use regular git cherry-pick locally.

[JAMIE]
So gh helps find; git helps apply.

[ALEX]
Exactly. At the Git CLI, you can cherry-pick a single commit. You can also cherry-pick a range of commits using the older-dot-dot-newer syntax, where the first endpoint is exclusive. And you can use no-commit if you want Git to apply the changes without creating the commit immediately, which gives you a chance to review or modify the result first.

[JAMIE]
That no-commit option sounds useful if I’m not totally sure the commit belongs exactly as-is.

[ALEX]
Yes. It gives you a checkpoint before the final commit is created.

[JAMIE]
What about conflicts?

[ALEX]
Cherry-pick can conflict for the same reason merges do: the target branch and that commit both touched overlapping code. Git pauses, marks the conflicting files, and you resolve them. In VS Code you can use the conflict editor. After resolving, stage the fixed files. Then continue the cherry-pick. If you decide this was a mistake, abort and Git returns you to where you started.

[JAMIE]
And for screen reader users, the appendix had a practical tip here.

[ALEX]
Yes: if a cherry-pick stops because of a conflict, run git status. It tells you which files need attention. The conflict markers in the files are plain text, so they read line by line in a straightforward way.

[JAMIE]
There was also a Copilot angle.

[ALEX]
Right. If you’re unsure whether a commit is the right one to cherry-pick, you can inspect it with git show and paste that output into Copilot Chat with a question like, “What does this commit do, and should I cherry-pick it onto my branch?” And if a conflict appears, Copilot can help explain the conflict or suggest a resolution.

[PAUSE]

[ALEX]
Now let’s move to interactive rebase, one of the most useful and most misunderstood Git tools.

[JAMIE]
This is the one people use to make their branch history look less chaotic before opening a pull request.

[ALEX]
Exactly. Interactive rebase lets you rewrite your local commit history before it becomes shared history. You can reorder commits, squash several tiny commits together, rewrite commit messages, drop a commit entirely, or pause to edit one.

[JAMIE]
Give me the golden rule first.

[ALEX]
Only rewrite commits that have not yet been pushed to a shared branch, unless you fully understand the consequences and are prepared to force push safely afterward. Rewriting public history can create problems for collaborators.

[JAMIE]
So interactive rebase is ideal for cleaning up your own branch before the branch becomes everybody else’s problem.

[ALEX]
Well said.

[JAMIE]
How do you start one?

[ALEX]
In VS Code, you’ll generally open the integrated terminal because full interactive rebase is really a terminal workflow. A common command is rebase interactively on HEAD tilde 3, which means, “open the last three commits for editing.” Another common form is interactive rebase against main, meaning, “rewrite everything on my branch since I diverged from main.”

[JAMIE]
And when that opens, what exactly am I looking at?

[ALEX]
A list of commits, one per line. Each line begins with a command word such as pick. Pick means keep the commit as-is. Squash means combine this commit into the one above it. Fixup also combines it into the one above but discards this commit’s message. Reword keeps the commit but asks you to edit the message. Drop removes the commit entirely. Edit pauses at that commit so you can amend it.

[JAMIE]
So the rebase file is really a little instruction sheet to Git.

[ALEX]
That’s a great mental model.

[JAMIE]
What’s a common cleanup example?

[ALEX]
Suppose you made one real commit and then two typo-fix commits. You can leave the first as pick and change the next two to squash. Save and close the file. Git then opens another editor asking for the final combined commit message. You write one clean message, save, and now those three commits become one tidy commit.

[JAMIE]
And reword is for when the code is fine but the message is vague.

[ALEX]
Exactly. Change pick to reword for that commit, save, and Git prompts you to edit the message.

[JAMIE]
The appendix also included an important editor tip.

[ALEX]
Yes. If rebase opens in vim and that’s not comfortable, especially for accessibility, set your Git editor to VS Code with core.editor set to code wait. Then future interactive rebases open in VS Code, which is often much more screen-reader-friendly.

[JAMIE]
What if I get nervous halfway through?

[ALEX]
Abort the rebase, and Git returns to the pre-rebase state. If you hit a conflict mid-rebase, resolve the conflict, stage the files, and continue the rebase. The pattern is similar to cherry-pick and merge: fix, stage, continue. If needed, abort.

[PAUSE]

[ALEX]
Next is git reset, which is all about undoing, but at different depths.

[JAMIE]
This is the one where the modes matter a lot.

[ALEX]
Very much. Reset can move HEAD and optionally change the staging area and working tree. The three classic modes are soft, mixed, and hard.

[JAMIE]
Let’s translate that into plain language.

[ALEX]
Soft reset moves the branch pointer back but keeps your changes staged. Mixed reset, which is the default, moves the branch pointer back and unstages the changes, but keeps them in your files. Hard reset moves the branch pointer back and discards the changes from both staging and working tree.

[JAMIE]
So soft is the gentlest undo, mixed is a deeper undo but keeps the work, and hard is destructive.

[ALEX]
Exactly. Hard reset is the dangerous one. The appendix explicitly warns that discarding changes with hard reset may be impossible to recover without reflog, and even then you’re depending on local history that may not last forever.

[JAMIE]
How might someone do this in VS Code?

[ALEX]
The appendix includes VS Code routes for undoing the last commit through the Source Control panel and through the Command Palette. Those are friendly entry points when you want to undo a recent commit without memorizing flags.

[JAMIE]
And at the CLI?

[ALEX]
There are several common patterns. Undo the last commit but keep changes staged: soft reset to HEAD tilde 1. Undo the last commit but keep changes unstaged: mixed reset to HEAD tilde 1, or simply reset HEAD tilde 1 since mixed is the default. Undo the last three commits but keep all changes unstaged: reset mixed to HEAD tilde 3. Undo the last commit and discard the changes entirely: hard reset to HEAD tilde 1. You can also reset to a specific commit SHA if you want to move directly to some earlier point.

[JAMIE]
There’s a separate but related task here too: unstaging.

[ALEX]
Yes. Sometimes you do not want to undo a commit at all. You simply staged the wrong file. In VS Code, you can unstage from the Source Control interface. In Git CLI, you can unstage a specific file by resetting that file, or unstage everything by resetting with no path. That removes content from the staging area but leaves the working tree unchanged.

[JAMIE]
That distinction is huge: unstaging is not deleting.

[ALEX]
Exactly. It just says, “don’t include this in the next commit yet.”

[PAUSE]

[ALEX]
Now let’s compare reset with revert, because people mix them up constantly.

[JAMIE]
Revert is the safe undo for shared branches.

[ALEX]
Right. Reset rewrites history. Revert preserves history by adding a new commit that undoes an earlier commit. That makes revert the safer choice when the branch is already shared, published, or protected.

[JAMIE]
So if I accidentally broke main and the bad commit is already on the remote, I should usually revert, not reset.

[ALEX]
Correct. In VS Code, there are interface-based ways to revert a commit. In the Git CLI, the standard form is revert followed by the commit SHA. By default that opens an editor so you can confirm or edit the revert commit message. If you want the default message without opening the editor, there’s a no-edit form. And if you want Git to apply the revert changes without committing yet, there’s a no-commit form so you can review or change the message yourself.

[JAMIE]
The appendix also mentions reverting a merge commit, which is a special case.

[ALEX]
Yes. Reverting a merge commit requires specifying which parent should be treated as the mainline. That’s because a merge has more than one parent, so Git needs to know which side you want to preserve when constructing the inverse change.

[JAMIE]
So when do I use revert versus reset in one sentence?

[ALEX]
Use reset for local history cleanup before sharing; use revert when history is already shared and you need a safe, auditable undo.

[PAUSE]

[ALEX]
Let’s talk about tags.

[JAMIE]
These are the labels you attach to important moments in history, like releases.

[ALEX]
Exactly. A tag marks a specific commit with a memorable name. There are two main kinds here: lightweight tags and annotated tags.

[JAMIE]
What’s the difference?

[ALEX]
A lightweight tag is basically just a named pointer to a commit. An annotated tag stores extra metadata like tagger, date, and a message, which is why annotated tags are generally recommended for releases.

[JAMIE]
How do you create them?

[ALEX]
In VS Code, the Command Palette can help create tags. With GitHub CLI, one convenient path is to create a release through GitHub’s API or release tooling, which can create both the underlying tag and a GitHub Release in one step.

[JAMIE]
That’s an important distinction: a Git tag and a GitHub Release are related but not identical.

[ALEX]
Right. The release is a GitHub platform object; the tag is a Git object. Some GitHub workflows create both together.

[JAMIE]
And the plain Git CLI options?

[ALEX]
Create a lightweight tag at the current commit with just the tag name. Create an annotated tag with the a flag and a message. Tag a specific older commit by giving the tag name and the commit SHA. You can list all tags, list tags matching a pattern, and show the details of an annotated tag.

[JAMIE]
Creating the tag locally doesn’t automatically publish it, though.

[ALEX]
Exactly. You then push either one specific tag or all local tags to GitHub. And if you need to delete a tag, there are two separate steps to think about: delete the local tag, and if it was already published, delete the remote tag on the GitHub remote as well.

[PAUSE]

[ALEX]
Now for detached HEAD.

[JAMIE]
This phrase sounds mysterious the first time you hear it.

[ALEX]
It does, but the core idea is simple: normally HEAD points to a branch name, and that branch name points to a commit. In detached HEAD, HEAD points directly to a commit instead of a branch. You can inspect history there, but if you make new commits and then move away without saving them onto a branch, they become easy to lose track of.

[JAMIE]
How do people end up there?

[ALEX]
Usually by checking out a specific commit SHA or sometimes a tag, instead of checking out a branch.

[JAMIE]
If I just want to get back to normal, what do I do?

[ALEX]
If you didn’t make any commits you care about, simply switch back to a real branch, often the branch you were on before.

[JAMIE]
And if I did make commits while detached?

[ALEX]
Then create a new branch at the current detached position. That anchors those commits to a branch, making them safe. After that, you’re on a real branch again and the work is no longer floating.

[JAMIE]
And VS Code can help show this state too.

[ALEX]
Yes. VS Code’s branch and source control indicators can reveal that you’re detached, and you can create or switch branches from the UI.

[PAUSE]

[ALEX]
Let’s tackle force pushing safely, because this is where good intentions can become team problems.

[JAMIE]
We need the difference between force and force-with-lease.

[ALEX]
Absolutely. Force push overwrites the remote branch with your local branch. Force-with-lease is the safer variant: it will refuse to overwrite the remote if someone else has pushed changes since your last fetch. So it acts like a seatbelt.

[JAMIE]
When would I need any force push at all?

[ALEX]
Most commonly after rewriting commit history on your own branch, especially after interactive rebase. Your local branch now has a different history than the remote branch, so a normal push is rejected. Force-with-lease is then the typical way to update the remote copy of your branch.

[JAMIE]
But not usually on shared branches like main.

[ALEX]
Correct. This is usually for your own feature branch, not protected shared branches.

[JAMIE]
How do the tools fit here?

[ALEX]
VS Code offers push flows, but for force pushing, the appendix explicitly says to use git directly. GitHub CLI likewise is not the place for the actual force-push flag; use Git CLI.

[JAMIE]
And the key commands?

[ALEX]
Safe force push is push with force-with-lease. There’s also a way to inspect what will happen before pushing, and there is unconditional force, plain force, which the appendix says to avoid unless you know exactly what you’re doing.

[JAMIE]
Can you walk through the common rebase-plus-force-push sequence?

[ALEX]
Sure. One: you’re on your feature branch and rebase it onto main to pick up the latest changes. Two: resolve any conflicts and continue the rebase. Three: because the remote still has the old pre-rebase history, you push your branch with force-with-lease so the remote branch is updated safely.

[JAMIE]
So the safe pattern is: rewrite locally, verify, then force-with-lease, not plain force.

[ALEX]
Exactly.

[ALEX]
Now let’s look at git bisect, which is one of Git’s most brilliant debugging tools.

[JAMIE]
This is the “find the exact bad commit” feature by binary search.

[ALEX]
Yes. Instead of checking every commit one by one, bisect repeatedly chooses a midpoint between a known bad commit and a known good commit. You test that midpoint and tell Git whether it’s good or bad. Each answer cuts the search space roughly in half.

[JAMIE]
How do you start?

[ALEX]
First, start bisect mode. Then mark the current broken state as bad. Then tell Git a commit you know was good, maybe by SHA, tag, or branch name from before the bug appeared.

[JAMIE]
After that, Git checks out a midpoint and asks me to judge it.

[ALEX]
Exactly. If the bug is present at that midpoint, mark it bad. If the bug is not present, mark it good. Git moves again. You keep repeating until Git identifies the first bad commit.

[JAMIE]
And when I’m done?

[ALEX]
End the bisect session to return to your original branch.

[JAMIE]
The appendix also mentions automation.

[ALEX]
Yes. If you have a reliable test script that exits successfully when things are good and fails when things are bad, you can ask Git to run that script automatically at each step. That turns bisect into a mostly hands-off search.

[JAMIE]
And there was a note later about Copilot helping if the bisect itself becomes confusing.

[ALEX]
Right. Copilot can help interpret odd outputs or help debug why your test script isn’t behaving consistently during bisect.

[PAUSE]

[ALEX]
Next up is git clean, which sounds mild but can be destructive.

[JAMIE]
This is for deleting untracked files, not tracked changes.

[ALEX]
Correct. Clean removes files Git is not tracking. That often means build artifacts, generated files, or scratch files cluttering your working directory. The big rule here is: always dry-run first.

[JAMIE]
Because otherwise you might delete something you forgot wasn’t tracked.

[ALEX]
Exactly. The dry-run form shows what would be deleted without actually deleting it. The appendix even notes an equivalent dry-run spelling, so learners recognize both forms.

[JAMIE]
What are the common variations?

[ALEX]
You can remove untracked files only, remove untracked files and directories, remove untracked plus ignored files—which is especially dangerous because that includes things listed in gitignore, like build outputs—and use interactive mode so Git prompts you before deleting items.

[JAMIE]
And the option meanings matter.

[ALEX]
Yes. The appendix explicitly explains the common flags: n for dry run, f for force, because Git usually requires force before deleting, d for directories, and x for ignored files too.

[JAMIE]
Is there a VS Code angle?

[ALEX]
There is mention of VS Code, but git clean is fundamentally more of a command-line task. The key teaching point is understanding what counts as untracked and using dry-run before any destructive clean.

[PAUSE]

[ALEX]
Now let’s talk about branch protection, because sometimes Git isn’t rejecting you due to a mistake in your command. Sometimes the repository rules are doing their job.

[JAMIE]
This is the “why can’t I push to main?” lesson.

[ALEX]
Exactly. Branch protection can block direct pushes, block merges until reviews pass, require status checks or CI, require up-to-date branches, and enforce other repository policies.

[JAMIE]
So if a push or merge is blocked, it may be policy, not a broken workflow.

[ALEX]
Right. And the correct flow for protected branches is very consistent. Never work directly on main. Create a branch. Make your changes and commit them. Push your branch, not main. Open a pull request on GitHub. Then wait for reviews and automated checks to pass, and merge through the pull request.

[JAMIE]
That line from the appendix is subtle but important: protection rules often apply to the merge into the protected branch, not to your ability to push your own feature branch.

[ALEX]
Exactly.

[JAMIE]
What if my branch is out of date with main and GitHub says it must be updated?

[ALEX]
You usually have two options. Option A is merge main into your branch. That creates a merge commit and is often straightforward. Option B is rebase your branch onto main. That gives a cleaner history, but because it rewrites your branch history, it usually requires a force-with-lease push afterward.

[JAMIE]
And those can be done in VS Code, plain Git CLI, or in some cases via GitHub tooling.

[ALEX]
Yes. The appendix includes VS Code and CLI approaches, and also notes GitHub CLI routes for updating a pull request branch from the GitHub side when that workflow is available.

[JAMIE]
How do I check what branch protection rules are actually active?

[ALEX]
Through GitHub CLI, you can query branch protection information for a branch like main. And on GitHub.com, repository settings expose the rules in the branch protection or ruleset interface.

[JAMIE]
So the real skill isn’t just knowing Git commands. It’s recognizing when repository governance is part of the workflow.

[ALEX]
Precisely.

Before we close, can we do a rapid decision guide?

[ALEX]

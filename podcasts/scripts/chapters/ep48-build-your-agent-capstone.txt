Before we get into the choices, what are learners supposed to walk away with?

[ALEX]
A full capstone workflow. You choose a repository, identify a real audience and workflow gap, create or improve an agentic asset, define its boundaries, test it locally or document testing, prepare a pull request or equivalent contribution package, and then either respond to review or explain what review you would seek.

[JAMIE]
And this challenge is the checkpoint, not the whole project.

[ALEX]
Right. The challenge is the evidence. The chapter is the project guide.

[JAMIE]
What counts as an acceptable contribution here?

[ALEX]
Several things. A new agent. An improved agent with better guardrails. A prompt or slash command. Repo-specific custom instructions. A skill or rule reference. A documentation update. Or even an issue triage or scoped proposal for the next agent the project should build.

[JAMIE]
So the common thread is usefulness, not format.

[ALEX]
Exactly. The evidence should show intent, scope, and practical value. For example, an agent file plus testing notes, a diff or pull request link, a prompt file with an example run, a custom instructions file with scope explanation, a skill draft with source notes, or an issue draft that clearly states mission, impact, and guardrails.

[PAUSE]

[JAMIE]
What should learners have in place before they start?

[ALEX]
They should already understand fork, branch, commit, push, and pull request workflow. They should have GitHub Copilot or Copilot Free active. They should have explored the Accessibility Agents repository, even if they choose a different capstone path. They should know which repository they want to help and why. And they should be able to explain the impact of their contribution in one or two sentences.

[JAMIE]
That last one matters because if you can't explain the value simply, the scope may still be fuzzy.

[ALEX]
Exactly right.

[JAMIE]
How long should this take?

[ALEX]
A workshop-ready capstone often takes about sixty to ninety minutes. A polished pull request can take longer, especially if the repo has tests, review rules, or unfamiliar architecture. And it's fully acceptable to finish the workshop with a strong draft and continue asynchronously.

[ALEX]
Now, there are three main contribution paths.

[JAMIE]
Let me guess. One is the default path, one is a smaller product-focused path, and one is bring your own repository.

[ALEX]
Perfect summary. The paths are Accessibility Agents, GLOW, or another repository you care about.

[JAMIE]
How should someone choose well instead of just choosing what sounds interesting?

[ALEX]
Ask five questions. First, impact: who benefits if this merges? Second, fit: does the repository already use agents, prompts, instructions, skills, or automation? Third, scope: can you make a useful change in one small pull request? Fourth, access: can you fork it or at least draft in your own fork? And fifth, safety: can you define guardrails that keep the asset from overreaching?

[JAMIE]
That safety question is easy to overlook when you're excited.

[ALEX]
Yes. A capstone contribution is not stronger because it sounds powerful. It's stronger when its boundaries are clear.

[PAUSE]

[ALEX]
Let's start with the default route: Accessibility Agents.

[JAMIE]
Why is that the default?

[ALEX]
Because it's built around exactly the kinds of assets you're learning to create: agents, prompts, instructions, skills, and accessibility workflows. It also has a large existing inventory, so learners can study patterns, copy structure, and make a focused contribution without inventing the whole ecosystem from scratch.

And the inventory is pretty substantial.

[JAMIE]
So if someone wants the most direct path to building a real agent, this is it.

[ALEX]
Exactly. It's a strong choice if you want clear examples, if you want your work to help accessibility reviewers, developers, document remediators, maintainers, or assistive technology users, and if you want the Challenge 16 autograder path that validates frontmatter, responsibilities, and guardrails.

[JAMIE]
What are good contributions there?

[ALEX]
A small specialist agent for a workflow that is not yet covered. Improving an existing agent's guardrails or output format. Updating stale counts, platform names, or release-note guidance in docs. Adding examples that explain how screen reader users, low-vision users, keyboard-only users, and sighted users should use an agent. Improving prompts so they produce more structured, reviewable output. Or sharpening a skill reference with clearer severity scoring and remediation guidance.

[ALEX]
The second route is GLOW.

[JAMIE]
And GLOW is more product-shaped, right?

[ALEX]
Yes. GLOW stands for Guided Layout and Output Workflow. It's a multi-surface accessibility toolkit for auditing, fixing, converting, and generating accessible documents. It spans web, desktop, command-line, VS Code agent tooling, and a Word add-in path.

[JAMIE]
So this is very document accessibility focused.

[ALEX]
Exactly. It centers document accessibility and large-print production. The source specifically notes alignment with the American Council of the Blind Large Print Guidelines, APH submission guidance, Microsoft Accessibility Checker style checks, and WCAG 2.2 AA concepts across formats like Word, Excel, PowerPoint, Markdown, PDF, EPUB, HTML, and CSS.

[JAMIE]
That's a lot of surfaces and standards.

[ALEX]
It is, and that's part of why a small, focused contribution can really matter. The source also listed a smaller current inventory than Accessibility Agents, with a handful of agent and prompt files, one instruction file, and one skill file, checked locally on May thirteenth, twenty twenty-six.

[JAMIE]
What kinds of things does GLOW actually do?

[ALEX]
It audits many document formats for accessibility issues. It can auto-fix some Word issues when the fix is deterministic, such as fonts, spacing, emphasis, headings, and margins. It generates ACB-compliant Word templates. It converts among formats using tools like MarkItDown, Pandoc, and LibreOffice. It supports multiple user surfaces. And it works with standards profiles such as ACB 2025 Baseline, APH Submission, and Combined Strict.

[JAMIE]
Why would a learner choose GLOW over Accessibility Agents?

[ALEX]
If they care about documents, large print, Office files, accessible publishing, or user workflows beyond traditional software development. Also if they want a smaller agent surface where one contribution can be very visible, or if they want to connect an agent to a real end-to-end product flow like upload, audit, fix, convert, template, export, and review.

[JAMIE]
What are good GLOW contributions?

[ALEX]
A routing agent that helps users choose between audit, fix, template, export, and convert workflows. Improved instructions for an existing large-print agent with clearer responsibilities and guardrails. A prompt for reviewing release notes against actual current features. Custom instructions that help contributors stay aligned with how GLOW works. Documentation that explains contributor workflows. Or a proposal for rule coverage the product should add next.

[JAMIE]
And there was also a safety note around GLOW.

[ALEX]
Yes, and it's important. In a workflow that touches accessibility remediation, agents should not overclaim certainty, silently modify standards-sensitive content without clear boundaries, or imply legal or compliance guarantees they cannot actually provide. Safer behavior is to explain scope, identify deterministic versus judgment-based fixes, preserve human review, and cite the standards profile or workflow being used.

[PAUSE]

[ALEX]
The third route is another repository.

[JAMIE]
This is the path for learners who already care about a project and want their capstone to land there.

[ALEX]
Exactly. But there are requirements. The repository should have a real audience and a plausible workflow gap. Your planned asset should fit the repo's structure and norms. The contribution should be small enough to review. You need a way to provide evidence even if you cannot merge directly, such as a fork, draft branch, issue proposal, or review-ready document. And you still need responsibilities and guardrails.

[JAMIE]
So even outside the default repositories, the same discipline applies.

[ALEX]
Yes. Good contributions in any repository include project-specific custom instructions, a maintainer-focused agent, issue templates, contribution documentation that is more accessible, prompts for release notes or triage, or a scoped plan for a future agent.

[JAMIE]
In other words, don't force an agent where a prompt, doc update, or instruction file would solve the problem better.

[ALEX]
Exactly.

[JAMIE]
Once a learner picks a path, what comes next?

[ALEX]
Define the mission. This is one of the most important steps.

[JAMIE]
Because if the mission is vague, everything downstream gets vague too.

[ALEX]
Correct. A strong mission explains who the asset is for, what workflow it helps with, what it should do, what it should refuse or defer, and what a successful output looks like. Ideally, you can say it in a few plain sentences.

[JAMIE]
Can we make that concrete?

[ALEX]
Sure. Instead of, this agent helps with accessibility, a better mission is, this agent reviews plain-language findings in accessibility audit notes, rewrites them for clarity, preserves technical meaning, and flags places where human reviewers should verify severity or standards references.

[JAMIE]
That already sounds testable.

[ALEX]
Exactly. Mission clarity supports testing, review, and safety.

[PAUSE]

[ALEX]
After the mission, you write or improve the agentic asset.

[JAMIE]
And the source was intentionally broader than just agent files.

[ALEX]
Yes. The asset might be an agent, prompt, custom instructions file, skill, workflow note, or documentation page. But because this chapter emphasizes agents, it spends special time on agent file structure.

[JAMIE]
Let's talk about location and naming first.

[ALEX]
Agent files should live where the repository expects them. Often that's a dot GitHub agents directory or another established pattern in the project. Naming should be descriptive and consistent, usually with the repo's existing conventions and, for agent files, the expected extension such as dot agent dot md.

[JAMIE]
So don't invent a random folder name or clever title if the repo already has a system.

[ALEX]
Right. Follow the house style.

[JAMIE]
And there was a minimal template idea too.

[ALEX]
Yes. At minimum, an agent asset should make its role obvious. It needs clear frontmatter or metadata expected by the platform, a concise description of responsibilities, explicit guardrails, and usable instructions for how it should respond.

[JAMIE]
The chapter included a named example, the Plain Language Findings Reviewer.

[ALEX]
It did, and that's a very useful model.

[JAMIE]
So what is that example trying to do?

[ALEX]
It's an agent focused on turning findings into plain-language, understandable output without distorting meaning. That means the responsibilities likely include rewriting for clarity, preserving essential technical content, organizing findings so they are reviewable, and helping users understand issues without unnecessary jargon.

[JAMIE]
And just as important, what should it not do?

[ALEX]
It should not invent facts, overstate certainty, claim a legal determination, or silently change the severity or standards basis of a finding without saying so. It should preserve the distinction between what is observed, what is inferred, and what needs human review.

[JAMIE]
That gets us into guardrails.

[ALEX]
Exactly. Guardrails are the boundaries that prevent an agent from becoming misleading or unsafe. They tell the agent when to stay in scope, when to ask for clarification, when to cite uncertainty, when to refuse, and how to present output responsibly.

[JAMIE]
And because this is an accessibility workflow, bad guardrails could create very polished but incorrect guidance.

[ALEX]
Yes, which is why the source emphasizes responsibilities and guardrails as first-class parts of the capstone, not optional polish.

[PAUSE]

[JAMIE]
What about YAML frontmatter fields?

[ALEX]
The exact fields can vary by environment and tooling, so learners should follow the authoritative docs and the host repository's examples. But the chapter clearly expects valid frontmatter that identifies the agent and supports discoverability and behavior. In practice, that usually means a name or title, a description, and any repository-specific metadata required by the platform or autograder.

[JAMIE]
So the safe rule is: copy the schema used by the target repository, don't improvise.

[ALEX]
Exactly. Especially if the project validates frontmatter automatically.

[JAMIE]
And the learning cards in this part were basically writing advice.

[ALEX]
Yes. Write the asset in plain, testable language. State responsibilities as actions. State guardrails as limits. Prefer specific output expectations over vague aspirations. And design for reviewability, so a maintainer can quickly see what the asset is for and where it might fail.

[PAUSE]

[ALEX]
Now let's move to testing locally.

[JAMIE]
This is the part people skip when they're eager to open the pull request.

[ALEX]
And it's one of the most valuable parts. Testing locally means running realistic prompts or scenarios against the asset before asking others to review it.

[JAMIE]
The source specifically mentions testing with GitHub Copilot Chat.

[ALEX]
Yes. The idea is to load or reference the agent in the environment it is meant for, then try tasks that match its mission. Give it clean cases, messy cases, ambiguous cases, and out-of-scope cases. Observe whether it follows its responsibilities and respects its guardrails.

[JAMIE]
So not just, did it produce words, but did it behave correctly.

[ALEX]
Exactly. A good testing checklist includes whether the asset is discoverable in the expected location, whether frontmatter is valid, whether the mission is understandable, whether outputs are structured and useful, whether the agent avoids overreach, whether it asks for clarification when needed, whether it handles inaccessible or ambiguous input safely, and whether its documentation tells contributors how to use and review it.

[JAMIE]
For blind and low-vision learners, I also want to call out something practical. Structured output is an accessibility issue too.

[ALEX]
Absolutely. If the output is easier to navigate with headings, bullets, short sections, and explicit labels, that helps reviewers using screen readers and everyone else. Accessibility applies to the contribution's behavior and its presentation.

[PAUSE]

[JAMIE]
Once the testing is decent, it's time to open or prepare the pull request.

[ALEX]
Right. Or, if you do not have direct repository access, prepare the equivalent review-ready package.

[JAMIE]
What should that include?

[ALEX]
A clear summary of what you changed and why. The audience or workflow gap you're addressing. Notes about responsibilities and guardrails. Testing notes or examples. Any limitations or open questions. And, if relevant, why you chose an agent versus a prompt, instruction file, or documentation update.

[JAMIE]
So the pull request description is doing teaching work for the maintainer.

[ALEX]
Exactly. A good PR description reduces reviewer effort.

[JAMIE]
The chapter also had a checklist here.

[ALEX]
Yes. Make sure the title is specific. Explain the problem and your solution. Link to any related issues or docs. Describe how you tested. Mention accessibility and safety considerations. Note if the contribution is draft status. And clearly mark anything you want reviewers to focus on.

[JAMIE]
And this is also where Challenge 16 evidence comes in.

[ALEX]
Correct. Evidence can be the pull request itself, the agent file, a documented contribution plan, or a review-ready draft. What matters is that the artifact shows clear purpose, responsibilities, guardrails, and enough substance to evaluate.

[PAUSE]

[ALEX]
After that comes review.

[JAMIE]
Sometimes the emotional hard part.

[ALEX]
Very true. Responding to review means staying specific, calm, and collaborative. Thank reviewers. Clarify intent when needed. Make small revisions. If you disagree, explain your reasoning with evidence and repository goals in mind. And if you haven't received review yet, describe the kind of review feedback you would actively seek.

[JAMIE]
Like asking, are the guardrails tight enough, is the scope too broad, are the outputs reviewable, does the naming fit the repo, and are the examples realistic?

[ALEX]
Exactly. Mature contributors don't just wait for review. They shape it.

[PAUSE]

[JAMIE]
How will learners know whether their capstone is strong?

[ALEX]
That's where the rubric helps. The core criteria are usefulness, scope, clarity, responsibilities, guardrails, testing, and review readiness.

[JAMIE]
Let's unpack that a little.

[ALEX]
Useful means it addresses a real need. Scoped means it is small enough to review and realistic for the workshop. Clear means the mission and intended audience are understandable. Responsibilities means the asset's job is explicit. Guardrails means limits and failure boundaries are explicit. Testing means you actually exercised the asset or documented how to do so. Review readiness means a maintainer could meaningfully evaluate it.

[JAMIE]
And what would exceed expectations look like?

[ALEX]
Thoughtful examples. Especially examples that include accessibility-specific user perspectives. Strong documentation. Clean, consistent naming and file placement. Excellent output structure. Clear source notes where standards or guidance matter. Honest discussion of limitations. And a contribution that anticipates reviewer questions before they are asked.

[JAMIE]
So exceeding expectations is not about making the biggest thing.

[ALEX]
Exactly. It's about making a well-judged, trustworthy thing.

[JAMIE]
What if someone gets stuck halfway through?

[ALEX]
The chapter gives a universal safety net. Reduce scope, don't abandon the project. If a full agent is too much, contribute a prompt, instructions draft, documentation page, issue proposal, or review-ready plan. If you can't finish testing, document the intended test cases and open questions. If you can't submit upstream, prepare the branch or draft in your fork.

[JAMIE]
So there is always a valid next smaller step.

[ALEX]
Yes. And that's an important professional habit. When blocked, produce the clearest useful artifact you can.

[JAMIE]
That is reassuring.

[ALEX]
It should be. The goal is not perfection on demand. The goal is a credible, helpful contribution process.
